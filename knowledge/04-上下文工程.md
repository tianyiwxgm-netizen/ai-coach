# 上下文工程：设计模型的信息环境

> **核心洞见：上下文工程的本质是一个优化问题——在LLM的注意力预算内，找到最小的高信号token集合，最大化期望结果的概率。你不是在"写提示词"，你是在策划一个精密的信息环境。**

---

## 一、从提示词工程到上下文工程

### 1.1 概念演进

2023-2024年，大家谈的是"提示词工程"——怎么写出更好的指令。2025年中，Andrej Karpathy在社交媒体上提出了一个更精确的术语：**上下文工程（Context Engineering）**。他指出，真正的技术活不是写一条提示词，而是设计模型在推理时能看到的整个信息环境。

Anthropic随后给出了正式定义：

> **上下文工程是在LLM推理期间策划和维护最优token集合的策略。**

这个定义有三个关键词：
- **策划（curating）**：不是堆砌信息，是精心筛选
- **维护（maintaining）**：不是一次性设置，是动态管理
- **最优（optimal）**：不是越多越好，是信噪比最高

为什么这比"提示词工程"更准确？因为在真实的AI应用中，发送给模型的上下文远不只是你写的那条提示词：

```
实际上下文 = 系统提示 + 对话历史 + 检索到的文档（RAG）
           + 工具描述 + 工具返回结果 + 结构化数据
           + 用户当前输入 + 格式约束 + Few-shot示例 + ...
```

提示词只占其中一小部分。上下文工程关注的是整个token集合的设计。

### 1.2 为什么这有效——注意力机制的物理约束

Transformer架构为n个token创建n²的注意力关系。这不是抽象概念，而是物理约束：

- **100个token**：10,000个注意力关系——模型能精确关注每个细节
- **10,000个token**：100,000,000个注意力关系——注意力开始稀释
- **100,000个token**：10,000,000,000个注意力关系——关键信息被淹没

这就是**Context Rot（上下文腐烂）**：上下文越长，模型准确回忆和利用信息的能力越低。不是线性下降，而是随token增加加速下降。

Anthropic的研究发现，LLM推理性能在约3000 tokens开始下降。大多数任务的最佳区间是150-300词。这不意味着你只能用300词，而是意味着**每一个额外token都必须值得它占用的注意力预算**。

**信噪比管理**就是这个优化问题的核心：在固定的注意力预算内，找到最小的高信号token集合。多一个token不加分，少一个token少关键信息——这个平衡点就是上下文工程要找的。

## 二、静态上下文设计

静态上下文是在请求发送前就确定的内容，不随用户输入变化。它是上下文工程的地基。

### 2.1 系统提示设计原则

系统提示有两个常见极端，都会失败：

```
极端A：过度硬编码
"当用户问X时返回Y，当用户问Z时返回W，格式必须是..."
→ 脆弱、无法泛化、维护成本高

极端B：过于模糊
"你是一个有帮助的助手，请尽力回答"
→ 没有任何约束力，等于没写
```

好的系统提示在两者之间：**定义身份、边界和行为原则，而非穷举具体场景。**

设计原则：
1. **先声明身份和目标**——模型需要知道"我是谁、要做什么"
2. **定义行为边界**——什么该做、什么不该做
3. **给出输出规范**——格式、风格、约束
4. **用正面指令而非否定指令**——"保持简洁"比"不要啰嗦"更有效
5. **按优先级排序**——模型对靠前内容的注意力更高

### 2.2 结构化组织：XML vs Markdown

结构化标签为注意力机制创建清晰的边界，让模型更容易区分不同类型的信息。

**XML风格**（Anthropic推荐，尤其适合Claude）：

```xml
<system>
<role>你是一个Python高级工程师，专注性能优化。</role>
<constraints>
- 只建议标准库或广泛使用的第三方库
- 每个建议必须附带性能影响的量化估算
- 不确定时明确说明
</constraints>
<output_format>
## 瓶颈分析
[按影响排序列出瓶颈]
## 优化建议
[每条建议包含：问题、方案、预期提升]
</output_format>
</system>
```

**Markdown风格**（通用，所有模型都能理解）：

```markdown
# 角色
Python高级工程师，专注性能优化。

# 约束
- 只建议标准库或广泛使用的第三方库
- 每个建议必须附带性能影响的量化估算

# 输出格式
按影响排序列出瓶颈，每条建议包含问题、方案、预期提升。
```

选择标准：Claude优先XML，GPT优先Markdown，混用也行。核心是**有结构比没结构好，一致比完美重要**。

### 2.3 Few-shot示例策略

示例是最强的上下文信号。一个精确的示例胜过十段描述。

**为什么示例比规则更有效？** 因为示例直接在注意力空间中创建了"Query-Key匹配模式"——模型可以直接模仿，不需要先"理解规则"再"转化为行为"。

设计原则：
- **质量优于数量**：2-3个精选示例 > 10个平庸示例
- **覆盖边界情况**：不要只给"正常"示例，给一个边界/异常示例
- **示例即规范**：示例中体现的格式、风格、深度就是模型会遵循的规范
- **负面示例慎用**：如果用，必须明确标注"这是错误示例"，否则模型可能模仿

```xml
<examples>
<example>
<input>分析函数：def get_user(id): return db.query(User).filter_by(id=id).first()</input>
<output>
## 瓶颈
1. **N+1查询风险**（影响：高）：若在循环中调用，每次触发独立SQL查询
## 优化建议
1. 批量查询：使用 `filter(User.id.in_(ids))` 替代循环单查，预期减少90%+数据库往返
</output>
</example>
</examples>
```

## 三、动态上下文管理（Agent场景）

静态上下文在简单问答中足够。但当模型需要在多步任务中自主行动（Agent场景），上下文管理变成一个动态工程问题。

### 3.1 JIT检索（Just-in-Time Context Retrieval）

核心理念：**不要预先加载所有可能需要的信息，而是在需要时精确获取。**

这类比人类认知——我们不会把整本手册背下来再开始工作。我们记住索引（"这个信息在哪里"），需要时去查。JIT检索就是让Agent做同样的事：存储轻量标识符，按需获取完整内容。

```
反模式：预加载一切
  启动时加载 → 所有文档（50K tokens）+ 所有API文档（30K tokens）
  → 上下文爆炸，信噪比极低

JIT模式：按需获取
  启动时加载 → 工具描述（500 tokens）+ 文档索引（200 tokens）
  执行时 → 根据当前步骤，检索所需的具体文档（2K tokens）
  → 上下文紧凑，信噪比极高
```

实现方式：
1. **工具化检索**：把知识库访问设计为工具，Agent在需要时调用
2. **分层索引**：先给摘要，Agent判断需要哪份完整文档再获取
3. **上下文窗口监控**：当token接近预算时，自动触发压缩或移除低优先级信息

### 3.2 工具描述设计

Anthropic提出工具设计三原则：

1. **Token高效**：描述尽量简洁，去掉冗余说明
2. **功能不重叠**：两个工具不该做同一件事，模型不应困惑该用哪个
3. **目的明确**：工具名称和描述要让模型一眼看懂何时该用

```
差的工具描述（61 tokens）：
name: search_database
description: 这个工具可以搜索数据库，它支持多种查询方式，
包括全文搜索、精确匹配、模糊匹配等。你可以用它来查找
用户信息、订单信息、产品信息等各种数据。

好的工具描述（22 tokens）：
name: search_users
description: 按姓名或邮箱查找用户。返回匹配的用户列表。
parameters: query (string, required): 搜索关键词
```

节省了2/3的token，信息量没有减少——反而更精确。

## 四、长时上下文策略

当对话或任务持续时间很长，上下文不可避免地增长。如何在不丢失关键信息的前提下控制上下文大小？

### 4.1 压缩（Compaction）

对话进行到一定长度时，用LLM自身来压缩历史：

```
原始对话历史（8000 tokens）：
  用户: [第1轮完整对话]
  AI: [第1轮完整回答]
  ... (20轮)

压缩后（800 tokens）：
  [对话摘要]
  - 用户正在构建一个FastAPI应用，主要目标是用户认证模块
  - 已确定使用JWT + Redis方案
  - 已完成：用户注册、登录接口
  - 待完成：Token刷新、权限中间件
  - 关键决策：选择RS256而非HS256，原因是支持多服务验证
  [最近3轮原始对话保留]
```

压缩策略：
- **选择性保留**：最近N轮保留原文，更早的轮次压缩为摘要
- **决策锚点保留**：关键决策及其原因永远保留原文
- **渐进式压缩**：越久远的信息压缩程度越高

### 4.2 结构化笔记（Agentic Memory）

比简单摘要更强的方案：让Agent维护一份结构化笔记，随任务推进持续更新。

```markdown
# Agent工作笔记（持续更新）

## 任务目标
构建用户认证模块，支持JWT + Redis

## 已完成
- [x] 用户模型定义（User, Role表）
- [x] 注册接口（POST /auth/register）
- [x] 登录接口（POST /auth/login）

## 进行中
- [ ] Token刷新机制

## 关键决策记录
| 决策 | 选择 | 原因 | 时间 |
|------|------|------|------|
| 签名算法 | RS256 | 支持多服务验证 | 第3轮 |
| Token存储 | Redis | 支持主动失效 | 第5轮 |

## 待解决问题
- 多设备同时登录如何处理？
```

这份笔记占用固定的token空间（通常500-1000 tokens），但携带了整个任务的关键上下文。每次LLM调用时注入这份笔记，而不是完整的对话历史。

### 4.3 子Agent架构

当单个Agent的上下文无法承载全部任务复杂度时，拆分为多个子Agent：

```
主Agent（编排者）
  ├── 子Agent A：代码实现（独立上下文）
  ├── 子Agent B：测试编写（独立上下文）
  └── 子Agent C：文档生成（独立上下文）
```

每个子Agent有自己的上下文窗口，只接收与其任务相关的信息。主Agent负责协调，传递必要的中间结果。这相当于**用架构手段解决上下文容量问题**。

## 五、RAG：检索增强生成

RAG是上下文工程中最重要的动态技术——它让模型能访问训练数据之外的知识。

### 5.1 基础RAG Pipeline

```
用户查询 → 查询编码（Embedding）→ 向量检索 → Top-K文档
    → 文档注入上下文 → LLM生成回答（基于检索到的文档）
```

核心组件：
1. **文档切分**：将长文档切成语义完整的chunk（通常200-500 tokens）
2. **向量化**：用Embedding模型将chunk编码为向量
3. **向量存储**：存入向量数据库（Pinecone、Weaviate、Chroma等）
4. **检索**：用户查询向量化后，在向量库中找最相似的chunk
5. **注入与生成**：将检索到的chunk注入上下文，LLM基于此生成回答

### 5.2 高级RAG技术

基础RAG的局限性催生了多种进阶方案：

**GraphRAG：** 不仅存储文档chunk，还构建实体之间的关系图。适合需要多跳推理的场景（"A公司的CEO是谁？他之前在哪家公司工作过？"——需要跨多个文档的实体关联）。

**SELF-RAG：** 模型自己决定是否需要检索、评估检索结果质量、判断自己的回答是否被检索内容充分支持。加入了自我反思循环，减少幻觉。

**RAFT（Retrieval-Augmented Fine-Tuning）：** 在微调阶段就加入检索文档，让模型学会从检索结果中提取答案、忽略干扰文档。相当于训练模型成为"开卷考试专家"。

**Long RAG：** 用更大的chunk（甚至整个文档）替代小chunk，减少信息碎片化。适合需要文档级理解的场景。

### 5.3 查询增强（Query Augmentation）

用户的原始查询往往不是最佳检索查询。查询增强技术弥补这个gap：

```
原始查询: "系统为什么这么慢？"

查询改写: "系统性能瓶颈 响应时间延迟 原因分析"

查询扩展（HyDE）: 先让LLM生成一个假想答案，
用假想答案去检索（因为答案与文档的语义距离更近）

多查询: 将一个查询拆成多个子查询并行检索，合并结果
```

### 5.4 重排序（Reranking）

向量检索是粗排——用语义相似度快速筛选候选。重排序是精排——用更复杂的模型（如Cross-Encoder）对候选文档重新评分：

```
向量检索 Top-20 → Cross-Encoder重排序 → Top-5注入上下文
```

重排序的成本高于向量检索，但精度大幅提升。实践中，先用向量检索召回较多候选（低成本），再用重排序精选最终结果（高精度）。

### 5.5 混合检索（Hybrid Retrievers）

单一检索方式各有盲区：

| 检索方式 | 优势 | 盲区 |
|---------|------|------|
| 向量检索（语义） | 理解语义相似性 | 精确关键词匹配差 |
| BM25（关键词） | 精确匹配强 | 不理解语义同义 |

混合检索结合两者：

```
查询 → 向量检索（语义匹配）──→ 融合排序（RRF等）→ 最终结果
    → BM25检索（关键词匹配）─┘
```

实践经验：混合检索几乎总是优于单一方式。推荐作为默认方案。

## 六、MCP协议（Model Context Protocol）

### 6.1 标准化连接

MCP是Anthropic于2024年底发布的开放协议，目标是标准化LLM与外部工具/数据源的连接方式。它解决的核心问题：每个AI应用都要自己实现与外部系统的集成，导致大量重复工作和不兼容的接口。

MCP的架构简洁：

```
AI应用（MCP Client）←→ MCP Server ←→ 外部系统（数据库、API、文件系统...）

MCP定义了三种能力：
1. Tools（工具）：模型可以调用的操作（如搜索、写入）
2. Resources（资源）：模型可以读取的数据（如文件、数据库记录）
3. Prompts（提示模板）：预定义的交互模式
```

### 6.2 2026年生态现状

截至2026年初，MCP已由Linux基金会治理，月SDK下载量超过9700万次。主要IDE（VS Code、Cursor、Windsurf）和AI平台（Claude、ChatGPT）均支持MCP。

MCP的意义在于将"上下文获取"标准化——开发者写一次MCP Server，所有支持MCP的AI应用都能使用。这大幅降低了上下文工程的集成成本。

## 七、知识库设计最佳实践

知识库是RAG和JIT检索的数据基础。设计不好的知识库会让整个上下文工程崩塌。

核心原则：

1. **原子性**：每个文档/chunk回答一个明确的问题，而非包含大量混杂信息
2. **自包含**：每个chunk不依赖上下文就能被理解（必要时重复背景信息）
3. **时效标注**：每个文档标注创建/更新时间，检索时可按时效过滤
4. **元数据丰富**：标签、分类、来源、权威度——元数据是精准检索的基础
5. **定期清理**：过时信息不删除会成为噪音，持续降低检索质量

```
差的知识库条目：
"关于部署的一些说明（更新于某个时间）：
首先确保Docker已安装，然后执行以下命令...
另外如果用K8s的话流程不同...
还有一个已知问题是..."

好的知识库条目：
标题: Docker单机部署步骤（v2.3）
更新时间: 2026-02-15
适用范围: 开发/测试环境，单节点部署
前置条件: Docker 24.0+, Docker Compose v2
步骤:
1. 克隆仓库: git clone ...
2. 配置环境变量: cp .env.example .env
3. 启动服务: docker compose up -d
4. 验证: curl http://localhost:8000/health
常见问题: [链接到独立文档]
```

## 九、刻意练习方案

### 练习1：信噪比感知（第1周）

**目标：** 训练识别上下文中的噪音token。

**方法：** 取你最近一次给AI的提示词（或系统提示），逐句标注：
- **H（高信号）**：删掉它模型一定会犯错
- **M（中信号）**：有帮助但不致命
- **L（低信号）**：删掉不影响结果

删掉所有L、一半M，测试输出质量是否下降。大多数情况下不会。

**进阶：** 对同一个任务，分别用500 tokens和2000 tokens的上下文完成，对比结果。记录"token效率"——用最少token达到同等质量的能力。

### 练习2：结构化改写（第2周）

**目标：** 把非结构化提示词转换为结构化上下文。

**方法：** 每天选一个你正在使用的AI交互场景，将其改写为XML/Markdown结构。对比改写前后的输出稳定性（同一个提示词执行5次，看结果一致性）。

### 练习3：JIT检索设计（第3周）

**目标：** 练习"索引而非加载"的思维。

**方法：** 选一个需要大量背景知识的任务（如代码审查、技术方案评审），设计信息获取流程：
1. 启动时只需要知道什么？（索引/摘要）
2. 执行每个步骤时需要获取什么？（按需检索）
3. 信息的获取方式是什么？（工具调用/RAG/文件读取）

画出信息流图，与"预加载一切"方案对比token使用量。

### 练习4：压缩与笔记（第4周）

**目标：** 练习长对话的上下文管理。

**方法：** 与AI进行一个复杂任务（如设计一个系统架构），故意让对话超过20轮。在第10轮和第20轮分别：
1. 写一份压缩摘要
2. 维护一份结构化笔记
3. 用摘要/笔记开启新对话，测试新对话是否能无缝继续

## 十、反模式：这样做会失败

### 反模式1：信息堆砌

```
"我把所有文档都塞进上下文，让模型自己找需要的"
→ Context Rot。信号被噪音淹没，模型反而更容易忽略关键信息。
```

**修正：** 精准检索 > 全量加载。宁可少给但给对的，不要多给但大部分无关。

### 反模式2：一劳永逸的系统提示

```
"我花了三天写了一个完美的系统提示，可以应对所有场景"
→ 过度硬编码。场景变了就失效，维护成本极高。
```

**修正：** 系统提示定义原则和边界，具体场景信息通过动态注入（RAG/工具）获取。

### 反模式3：忽略上下文的时序性

```
"对话历史全部保留，从不压缩"
→ 早期对话中已被推翻的决策仍然影响模型，产生矛盾输出。
```

**修正：** 定期压缩，保留最终决策而非决策过程。用笔记记录"最新结论"，而非保留"所有讨论"。

### 反模式4：工具描述膨胀

```
"给Agent注册了30个工具，每个都有详细描述"
→ 工具描述本身消耗大量token，模型在选择工具时犹豫不决。
```

**修正：** 遵循Anthropic的工具设计三原则——token高效、功能不重叠、目的明确。如果工具超过10个，考虑分层（按场景动态加载工具子集）。

### 反模式5：RAG即插即用

```
"接入了RAG，效果一般，可能是模型不够好"
→ 大概率是检索质量差、chunk切分不合理、或注入方式粗暴。
```

**修正：** RAG效果80%取决于检索质量，20%取决于生成。先优化文档切分、Embedding选择、重排序，再考虑换模型。

## 十一、专家怎么说

### Andrej Karpathy——概念提出者

Karpathy明确区分了上下文工程与提示词工程："上下文工程比提示词工程更重要。提示词只是上下文的一个子集。真正的功夫在于设计模型能看到的整个信息环境——包括什么时候检索什么信息、怎么压缩历史、怎么设计工具。"

他的Software 3.0框架将这个理念推到极致：在Software 3.0中，英语（自然语言）就是编程语言，上下文就是运行时环境。你构建的不是代码，而是一个让LLM能正确运行的信息空间。

### Anthropic工程团队——系统化实践

Anthropic在其博客中给出了关于上下文工程最有操作性的定义和指南。核心观点：

- "找到最小的高信号token集合，最大化期望结果的概率"——这是上下文工程的优化目标
- 上下文不是静态的文本块，而是需要"策划和维护"的动态系统
- 工具描述是被严重低估的上下文组成部分——差的工具描述会让Agent在每一步都浪费注意力

他们的工具设计实践：每个工具描述控制在3行以内，工具名称用动词+名词（如`search_users`而非`user_tool`），参数描述只包含模型需要知道的约束。

### Simon Willison——实践主义者

Willison在多年LLM实践后得出结论："使用LLM获得好结果的大部分技巧归结为管理其上下文——当前对话中的文本。" 他强调这是一种需要大量实践才能掌握的技能，不是读几篇文章就能学会的。

他的具体实践包括：
- 每次对话都有意识地监控上下文中有什么
- 新对话而非继续旧对话——保持上下文干净
- 把关键信息以结构化方式放在对话开头

### 行业趋势数据

Gartner预测到2026年底，40%的企业应用将包含AI Agent。这意味着上下文工程不再是"高级技巧"——它是每个开发者构建AI功能时的基础能力。当Agent成为标配，上下文管理的质量直接决定产品可靠性。
