1. 根据用户任务描述匹配下方「场景映射表」
2. 读取对应方法论文件，提取「核心要点」和「实践建议」
3. 写入指导报告的「推荐方法论」章节，引用文件路径
4. 未匹配时使用通用推荐：`fm-方法论决策树.md` + `00-北极星.md`

## 教练使用指南

教练流程遵循 `common/COACH_TEMPLATE.md` 定义的三阶段结构，知识库在每个阶段的使用方式不同。

### Phase 1: 理解（1-2 轮）

**目标**：快速掌握任务全貌，判断复杂度。

- **Round 1 — 任务快照**：收集 3 个核心信息（任务目标、当前状态、现有资料）
- **Round 2 — 复杂度判断**：根据 Round 1 回答判定复杂度等级
  - **简单**（CRUD/配置/单点修复）→ 跳过 Phase 2，直接进入 Phase 3
  - **中等**（多模块/有集成/有现有代码）→ Phase 2 问 1 轮
  - **复杂**（跨系统/架构变更/新领域）→ Phase 2 问 2 轮

**知识库用法**：此阶段暂不检索知识库，专注倾听用户。仅在需要辅助判断复杂度时参考 `00-北极星.md` 中的复杂度阶梯。

### Phase 2: 诊断（1-2 轮，自适应）

**目标**：定位关键信息缺口，为方法论推荐做准备。每轮最多 3 个问题。

- 根据用户描述的任务，查下方「场景映射表」找到对应文件
- 从文件中提取 2-3 个与当前任务最相关的要点
- 用这些要点构造诊断问题（"你有没有考虑过 X？"）
- 问题由角色 SKILL 的问题池提供，按复杂度等级选取数量

**退出条件**（满足任一即结束）：
- 已收集足够信息可推荐方法论
- 用户表示"差不多了"/"就这些"
- 已达最大轮数（简单 0 轮，中等 1 轮，复杂 2 轮）

### Phase 3: 方法论指导报告（1 轮）

**目标**：输出结构化的方法论指导报告，不给具体方案。

- 推荐 1-3 个方法论，每个引用具体文件路径（如 `knowledge/03-提示词工程.md`）
- 从对应文件提取核心要点（3-5 个）写入报告
- 如果用户是新手（参考 `02-心智模型.md` 的跃迁阶段），用更通俗的语言
- 必须包含「示范提示词」— 用户可直接在新会话中使用
- 必须包含「执行建议」— 推荐 superpowers 标准流程（brainstorming → writing-plans → TDD → code review → verification）

**硬性约束**：
- 不产出表结构设计、API 接口规格、代码架构图、具体代码
- 只推荐方法论 + 为什么适用 + 怎么用
- 只给示范提示词（让用户在新会话中让 AI 去执行）

### 在快速提问模式中

- 直接从相关文件提取答案
- 引用具体章节，方便用户深入阅读
