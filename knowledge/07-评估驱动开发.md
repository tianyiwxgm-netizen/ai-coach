# 评估驱动开发：用科学方法证明AI系统有效

> **核心洞见：评估驱动开发的本质是将AI系统的改进从"凭感觉调参"转变为"用数据决策"。先定义"什么是好的"，再构建系统，持续用评估闭环驱动迭代——这是区分业余者和专业者的分水岭。**

---

## 一、为什么评估是第一公民

### 1.1 Andrew Ng的核心判断

Andrew Ng在与数十个AI团队合作后，给出了一个极具预测力的观察：

> "与很多团队合作后发现，能否驱动严格的eval和error analysis流程，是预测一个人能否做好Agent系统的最强单一指标。"

这不是"最好也做做评估"级别的建议。这是"没有评估能力就做不好AI系统"级别的断言。

为什么如此绝对？因为AI系统有一个传统软件没有的根本特征：**非确定性**。同样的输入可能产生不同的输出，而"看起来合理"不等于"真的正确"。没有系统化的评估，你对系统质量的判断就是在猜。

### 1.2 类比TDD：先定义好坏，再开始构建

评估驱动开发与测试驱动开发（TDD）的哲学一脉相承：

```
TDD:   先写测试（定义期望行为）→ 再写代码（实现行为）→ 运行测试（验证行为）
EDD:   先写评估（定义好的输出）→ 再写提示/系统（产出输出）→ 运行评估（度量质量）
```

区别在于，TDD的测试是二值的（pass/fail），而EDD的评估是连续的（0-100分，或多维度评分）。AI系统的"正确"往往不是黑白分明的，你需要在灰度中建立标准。

### 1.3 底层原理：为什么有效

评估驱动开发有效的根本原因有三个：

**消除认知偏差。** 人类对AI输出的判断有强烈的确认偏误——你倾向于认为"还不错"，因为AI输出的文本流畅且自信。系统化评估用标准化指标替代主观感受。

**积累可比较的基准。** 没有评估，你无法回答"改了提示词之后到底变好了还是变差了"。每次修改都是盲人摸象。有了评估，每次修改都有数据支撑。

**加速迭代循环。** 当你有自动评估体系，实验的周期从"手动检查一批输出花半天"变成"跑一次评估脚本花5分钟"。速度提升一个数量级，迭代次数也会提升一个数量级。

LangChain的Terminal Bench案例是最有力的证据：他们的Agent排名从Top 30跳到Top 5（52.8% → 66.5%），仅靠改进治具，模型完全没换。这种精准优化只有在严格评估闭环下才可能实现——你必须知道哪里差、差多少、改了之后好了多少。

## 二、评估体系设计

### 2.1 七个评估维度

不同的AI应用关注不同的维度，但以下七个维度构成一个完整框架：

| 维度 | 衡量什么 | 典型指标 | 适用场景 |
|------|---------|---------|---------|
| **准确性** | 输出是否事实正确 | 精确率、召回率、F1 | 问答、信息抽取 |
| **相关性** | 输出是否回答了用户的问题 | 1-5分评分、命中率 | 搜索、RAG、对话 |
| **完整性** | 是否覆盖了所有要点 | 要点覆盖率 | 总结、报告生成 |
| **格式合规** | 是否符合指定的输出格式 | JSON解析成功率、字段完整度 | 结构化输出 |
| **安全性** | 是否包含有害/敏感内容 | 拒绝率、泄漏率 | 所有面向用户场景 |
| **延迟** | 响应时间是否可接受 | P50/P95/P99延迟 | 实时交互 |
| **成本** | 每次调用的token消耗和费用 | $/请求、tokens/请求 | 大规模部署 |

**关键原则：不要试图同时优化所有维度。** 先确定你的场景中哪2-3个维度最关键，集中评估和优化这些。

### 2.2 三种评估方法

```
评估方法光谱：
  自动评估 ←───── LLM-as-Judge ─────→ 人工评估
  (快、便宜、刚性)                    (慢、贵、灵活)
```

**自动评估：** 用规则或代码判断。适合有明确正确答案的场景。
- 精确匹配：输出是否等于期望答案
- 正则/模式匹配：是否符合格式要求
- 代码执行：生成的代码能否运行、测试能否通过
- 数值指标：BLEU、ROUGE等文本相似度

**LLM-as-Judge：** 用另一个LLM评估输出质量。适合需要语义判断的场景。
- 用强模型评估弱模型的输出
- 提供明确的评分标准（rubric）
- 注意偏差：LLM评判器倾向于给长答案打高分、偏好自己生成风格的内容
- 建议多次评估取平均，或使用多个评判器

**人工评估：** 人类专家打分。适合高风险、主观质量判断、评估体系初建阶段。
- 成本高但最可靠
- 用于校准自动评估和LLM-as-Judge的准确性
- 建议建立标注指南，保证评估者之间的一致性

**实用策略：从人工评估开始，逐步用自动评估替代。** 先让人类标注一批"黄金标准"，然后训练/调试自动评估使其与人类判断对齐。

### 2.3 评估集设计

评估集是你的"试卷"，它的质量直接决定评估的可信度。

**代表性：** 覆盖真实使用场景的分布。如果80%的用户查询是简单问题，你的评估集中也应该80%是简单问题——否则你优化的方向会偏离真实需求。

**边界案例：** 有意设计"刁钻"的输入。模糊指令、矛盾信息、超长输入、多语言混合、格式不规范。系统在边界案例上的表现往往暴露真正的问题。

**对抗样本：** 尝试让系统犯错的输入。提示注入、引导幻觉、试探安全边界。这些是上线前必须通过的"压力测试"。

**数量级参考：**
- 原型验证阶段：20-50条
- 开发迭代阶段：100-500条
- 生产监控阶段：500+条，持续从真实数据补充

### 2.4 基准测试的正确使用

基准测试（Benchmark）是行业通用的评估标准，如HumanEval（代码）、MMLU（知识）、MT-Bench（对话）。

**正确用法：**
- 用于模型选型时的横向对比
- 用于跟踪行业水平线
- 作为内部评估的参考维度之一

**常见误用：**
- 用通用基准得分推断特定场景的表现（你的任务不是HumanEval）
- 过度优化基准分数（Goodhart定律：指标变成目标时就不再是好指标）
- 忽视任务特异性评估，只看通用排行榜

**Andrew Ng的建议：** 先用最好的模型做出来，只在成本成为问题时才优化模型选择。不要花太多时间在模型对比基准上——构建你自己的任务专属评估集，价值比任何公开基准都大。

## 三、Error Analysis方法论

如果说评估告诉你"系统表现如何"，那么Error Analysis告诉你"为什么不好"和"怎么改进"。这是从评估到改进的桥梁。

### 3.1 系统化错误分类

拿到一批失败案例后，不要逐条修复，先分类：

```
错误分类体系：
├─ 检索错误（信息不在上下文中）
│   ├─ 召回失败：相关文档没被检索到
│   └─ 精度失败：检索到的文档不相关
├─ 理解错误（信息在上下文中但未正确理解）
│   ├─ 指令遗漏：模型忽略了某条指令
│   └─ 语义误解：模型误解了输入含义
├─ 推理错误（理解正确但推理过程有误）
│   ├─ 逻辑谬误：推理链中某步不成立
│   └─ 过度推断：在信息不足时编造结论
├─ 生成错误（推理正确但输出有问题）
│   ├─ 格式错误：输出不符合要求的格式
│   ├─ 冗余/缺失：输出过长或遗漏关键信息
│   └─ 幻觉：生成了不存在的事实
├─ 工具错误（工具调用出问题）
│   ├─ 选错工具：不该调用的工具被调用
│   ├─ 参数错误：工具参数不正确
│   └─ 结果误用：工具返回正确但解读错误
└─ 安全错误
    ├─ 提示注入：被用户输入劫持
    └─ 信息泄漏：暴露了不该暴露的信息
```

### 3.2 根因分析

分类后，对每类错误追问根因：

| 错误类型 | 根因追问 | 典型修复方向 |
|---------|---------|------------|
| 检索失败 | 是embedding模型的问题还是分块策略的问题？ | 调整chunk size、改进检索策略 |
| 指令遗漏 | 指令太多导致注意力稀释？还是指令不够清晰？ | 精简指令、结构化提示、XML标签分隔 |
| 推理错误 | 是任务本身超出模型能力？还是缺少引导？ | 添加CoT、分解子任务、升级模型 |
| 格式错误 | 是提示词中格式要求不够明确？ | 添加输出示例、使用结构化输出API |
| 工具错误 | 是工具描述不够清晰？还是工具功能有重叠？ | 改进工具描述、减少工具数量、添加使用示例 |

### 3.3 错误优先级排序

不是所有错误都值得修。用影响矩阵排序：

```
          高频率
            │
   优先修复  │  重点关注
  (高频高影响) │ (高频低影响)
            │
  ──────────┼──────────── 高影响
            │
   观察监控  │  记录但不急
  (低频高影响) │ (低频低影响)
            │
          低频率
```

**一个实用法则：** 先花1小时把所有错误按类别计数。通常你会发现，80%的错误集中在2-3个类别。专注修复这2-3个类别，就能获得最大改善。

## 四、A/B测试与实验

### 4.1 提示词A/B测试

当你修改了提示词，不要只跑几个例子"看看感觉"。用评估集做正式对比：

```python
# 伪代码：提示词A/B测试框架
def ab_test_prompts(prompt_a, prompt_b, eval_set, evaluator):
    results_a = [run_and_evaluate(prompt_a, case, evaluator) for case in eval_set]
    results_b = [run_and_evaluate(prompt_b, case, evaluator) for case in eval_set]

    print(f"Prompt A 平均分: {mean(results_a):.2f}")
    print(f"Prompt B 平均分: {mean(results_b):.2f}")

    # 检查统计显著性
    t_stat, p_value = ttest_ind(results_a, results_b)
    print(f"p-value: {p_value:.4f}")
    if p_value < 0.05:
        print("差异具有统计显著性")
    else:
        print("差异不显著，可能是随机波动")
```

### 4.2 避免的陷阱

**样本量不足。** 5个例子的"A比B好"毫无意义。至少用50-100个评估用例，才能对结论有基本信心。

**多重比较问题。** 如果你同时测了10个提示词变体，其中"最好的"可能只是运气好。测试变体越多，显著性阈值应越严格。

**评估集泄漏。** 不要用评估集中的案例作为Few-shot示例放进提示词。这相当于"背题"，评估分数虚高，上线后表现会回落。

**忽视回归。** 改进了某类问题，可能恶化了另一类。每次变更都要在完整评估集上测试，不只是新增的案例。

## 五、RAG评估专题

RAG（检索增强生成）系统有特殊的评估需求，因为错误可能出在检索阶段，也可能出在生成阶段。RAGAS框架提供了四个核心指标来精确定位问题：

### 5.1 四个核心指标

```
                    检索质量                        生成质量
              ┌───────────────┐              ┌───────────────┐
              │ Context       │              │ Faithfulness  │
              │ Precision     │              │ (忠实度)       │
              │ (上下文精度)   │              │               │
              │               │              │ 生成的答案是否  │
              │ 检索到的文档中  │              │ 忠于检索到的    │
              │ 有多少是相关的  │              │ 上下文内容      │
              ├───────────────┤              ├───────────────┤
              │ Context       │              │ Answer        │
              │ Recall        │              │ Relevancy     │
              │ (上下文召回)   │              │ (答案相关性)    │
              │               │              │               │
              │ 回答问题所需的  │              │ 生成的答案是否  │
              │ 信息是否都被    │              │ 回答了用户的    │
              │ 检索到了       │              │ 问题           │
              └───────────────┘              └───────────────┘
```

### 5.2 诊断矩阵

根据四个指标的高低组合，可以精确定位问题所在：

| Context Precision | Context Recall | Faithfulness | Answer Relevancy | 诊断 | 修复方向 |
|:-:|:-:|:-:|:-:|------|---------|
| 低 | - | - | - | 检索噪音大 | 优化embedding、改进rerank |
| - | 低 | - | - | 关键信息未检索到 | 扩大检索范围、改分块策略 |
| 高 | 高 | 低 | - | 检索到了但生成时"编造" | 强化"仅基于上下文回答"的指令 |
| 高 | 高 | 高 | 低 | 答案正确但没回答用户问题 | 优化生成提示词，强调回答问题 |
| 高 | 高 | 高 | 高 | 系统表现良好 | 持续监控，扩充评估集 |

这个诊断方法的价值在于：**不需要猜"是检索的问题还是生成的问题"，指标会直接告诉你。**

## 六、Agent评估

Agent系统比单次LLM调用复杂得多，评估也更具挑战。以下是四个关键维度：

### 6.1 任务完成率

最直接的指标：Agent是否完成了任务？

```
任务完成率 = 成功完成的任务数 / 总任务数

细化：
- 完全完成：100%达到预期结果
- 部分完成：核心目标达成，细节有偏差
- 失败：未完成或结果完全错误
- 超时：在时间限制内未完成
```

### 6.2 步骤效率

完成了任务不够，还要看效率：

```
步骤效率指标：
- 平均步骤数：完成任务用了几步？（越少越好）
- 冗余步骤率：有多少步是不必要的？
- 回退率：需要推翻前序步骤重来的比例
- 死循环率：陷入重复操作的频率
```

**一个合格的Agent系统应该在同类任务上步骤数稳定。** 如果方差很大，说明规划能力不稳定。

### 6.3 工具使用合理性

Agent选对工具了吗？参数给对了吗？

```
评估维度：
- 工具选择准确率：该用A工具时是否用了A
- 参数正确率：调用参数是否正确
- 工具调用时机：是否在合理的时间点调用
- 不必要调用率：有多少工具调用是多余的
```

### 6.4 错误恢复能力

Agent遇到错误时的表现，是区分"演示级"和"生产级"系统的关键：

```
测试场景：
- 工具返回错误：Agent能否识别错误并重试/换工具？
- 中间结果异常：Agent能否检测异常并调整策略？
- 上下文不足：Agent能否主动获取更多信息？
- 死路回退：走进死胡同后能否回到正确路径？
```

**建议：在评估集中故意注入异常场景（工具超时、返回空结果、格式不对），测试Agent的鲁棒性。**

### 7.1 评估计划模板

```markdown
# 评估计划：[项目/功能名称]

## 1. 评估目标
- 核心问题：我们要证明什么？
- 成功标准：什么指标达到什么水平算"可上线"？

## 2. 评估维度（按优先级排序）
| 维度 | 权重 | 目标值 | 当前值 | 评估方法 |
|------|------|--------|--------|---------|
| 准确性 | 40% | >90% | - | 自动评估 |
| 相关性 | 30% | >4.0/5 | - | LLM-as-Judge |
| 延迟 | 20% | P95<3s | - | 自动监控 |
| 安全性 | 10% | 0误判 | - | 对抗测试 |

## 3. 评估集
- 总量：200条
- 来源：生产日志采样(60%) + 手工构造边界案例(30%) + 对抗样本(10%)
- 更新频率：每两周补充新案例

## 4. 评估流程
- 每次提示词/策略变更 → 全量评估集跑一遍
- 每周 → 用生产真实数据补充评估集
- 每月 → 人工审查评估集的代表性

## 5. 负责人与时间线
- [姓名] — [日期]
```

## 八、刻意练习方案

### 练习1：为现有项目建立评估集（2小时）

**步骤：**
1. 选择你正在使用AI的一个场景（如代码生成、文档总结、客服回复）
2. 从真实使用记录中挑选30个输入
3. 为每个输入手写一个"理想输出"或评分标准
4. 跑一次评估，记录基准分数
5. 修改提示词，再跑一次，对比分数

**成功标准：** 你能用一个数字（而非直觉）描述系统质量。

### 练习2：Error Analysis实战（1小时）

**步骤：**
1. 从练习1的评估结果中，找出所有得分低于3的案例
2. 逐条分类错误类型（用3.1节的分类体系）
3. 统计每类错误的数量
4. 对最大类别进行根因分析
5. 提出一个修复方案并验证

**成功标准：** 你能用数据说出"系统最大的问题是X类错误，占比Y%，根因是Z"。

### 练习3：搭建LLM-as-Judge流程（3小时）

**步骤：**
1. 使用7.3节的模板创建评判提示词
2. 选择一个强模型作为评判器
3. 让评判器和人类分别评估同一批输出
4. 计算评判器与人类的一致率
5. 根据不一致的案例调整评判提示词
6. 迭代直到一致率 > 80%

**成功标准：** 你拥有一个自动评估流程，其判断与你的人工判断高度一致。

### 练习4：A/B测试实战（2小时）

**步骤：**
1. 选择一个你想优化的提示词
2. 写出2-3个变体
3. 在评估集上分别运行所有变体
4. 对比得分，检查差异是否显著
5. 选择最优变体，记录原因

**成功标准：** 你能用p-value而非直觉判断"哪个提示词更好"。

## 九、反模式：这样做会失败

### 反模式1："看看就行"综合症

**表现：** 改了提示词后，随便试两三个例子，"感觉还行"就上线。

**为什么失败：** 2-3个例子的统计意义为零。你可能恰好试到了容易的案例。上线后遇到评估集外的情况立刻暴露问题。

**修正：** 建立最小评估集（至少20条），每次变更必须跑完整评估。

### 反模式2：指标至上

**表现：** 过度优化某个评估指标，忽视了用户体验。评估分数越来越高，但用户满意度没有提升。

**为什么失败：** Goodhart定律——当指标变成目标时，它就不再是好指标。你可能在优化评估集中的模式，而非真正的质量。

**修正：** 定期用真实用户数据刷新评估集。评估指标与用户满意度/业务指标做关联分析。

### 反模式3：评估集一成不变

**表现：** 建了评估集后再也不更新。系统在旧评估集上分数越来越高，但对新类型的输入表现差。

**为什么失败：** 真实世界的输入分布在变化。用户会用你没想到的方式使用系统。

**修正：** 每周/每两周从生产日志中采样补充评估集。特别关注失败案例和边界案例。

### 反模式4：跳过Error Analysis

**表现：** 看到评估分数就直接改提示词。"分数低？那我加更多指令试试。"

**为什么失败：** 不做根因分析的修复是盲目的。你可能在"加更多指令"导致上下文膨胀、注意力稀释，反而让其他维度变差。

**修正：** 每次评估后必须做Error Analysis。先分类错误、找根因，再针对性修复。

### 反模式5：只在开发阶段评估

**表现：** 开发时建了完善的评估体系，上线后就不管了。

**为什么失败：** 模型API可能更新（即使版本号不变，provider也可能微调）、用户使用模式在变化、数据分布在漂移。OpenAI官方明确建议：生产环境锁定模型版本号，建立衡量提示词性能的eval。

**修正：** 部署持续监控，定期用生产数据跑评估，设置质量下降告警。

## 十、持续改进循环

### 10.1 版本控制提示词和策略

提示词是代码。用版本控制管理它：

```
prompts/
├── v1.0_baseline.txt          # 初始版本
├── v1.1_add_cot.txt           # 添加CoT
├── v1.2_fix_format.txt        # 修复格式问题
├── v2.0_restructure.txt       # 重构提示词结构
├── CHANGELOG.md               # 每个版本的变更原因和评估结果
└── eval_results/
    ├── v1.0_results.json
    ├── v1.1_results.json
    └── ...
```

每个版本记录：变更了什么、为什么变更、评估结果如何。这样你可以随时回滚，也能从历史中学习"什么类型的修改通常有效"。

### 10.2 回归测试

每次修改提示词后，不仅测新增/修复的案例，还要测所有旧案例。确保修复A不破坏B。

```
回归测试流程：
1. 修改提示词
2. 在完整评估集上运行
3. 与上一版本对比
4. 如果某些维度下降 > 2%，审查原因
5. 确认总体改善后再部署
```

### 10.3 生产监控与反馈收集

```
               生产环境
                  │
    ┌─────────────┼─────────────┐
    │             │             │
  日志采集      用户反馈      质量采样
    │             │             │
    ▼             ▼             ▼
  Langfuse     反馈按钮      定期人工
  可观测性     (👍/👎)       审查输出
    │             │             │
    └──────┬──────┘             │
           │                    │
           ▼                    │
     自动告警                   │
  (质量低于阈值)                │
           │                    │
           ▼                    ▼
      补充评估集 ←───────── 新的Error Analysis
           │
           ▼
      提示词迭代
```

**Langfuse等可观测性工具的价值：** 它们记录每次LLM调用的输入、输出、耗时、token数和成本。当系统出问题时，你可以回溯具体是哪次调用、什么输入导致了什么输出。这对生产级AI系统是必备基础设施。

### 10.4 商业指标对接

技术指标只有映射到商业指标才有真正的价值：

| 技术指标 | 商业指标 | 映射逻辑 |
|---------|---------|---------|
| 准确率提升5% | 客服工单减少X% | 回答更准确 → 用户自助解决率提升 |
| 延迟降低1s | 转化率提升Y% | 响应更快 → 用户体验更好 → 留存提升 |
| 幻觉率降低 | 投诉率降低 | 错误信息减少 → 用户信任提升 |
| 成本/请求降低 | 利润率提升 | 直接成本降低 |

**ROI计算框架：**
```
AI系统ROI = (带来的价值 - 系统成本) / 系统成本

带来的价值：
- 效率提升：节省的人力成本
- 质量提升：减少的错误成本
- 规模效应：人工无法处理的量

系统成本：
- API调用费用
- 开发和维护人力
- 评估和监控投入
```

## 十一、专家怎么说

**Andrew Ng** 给出了最根本的定位：

> "与很多团队合作后发现，能否驱动严格的eval和error analysis流程，是预测一个人能否做好Agent系统的最强单一指标。"

他同时给出了务实的建议：先用最好的模型做出来，只在成本成为问题时才优化模型选择。这意味着：**不要在模型选择上纠结，把精力放在评估和迭代上。**

**Simon Willison** 从独立开发者角度给出了最直接的警告：

> "你绝对不能外包给机器的一件事就是测试代码是否真的有效。"

AI可以生成代码、生成测试用例，但**判断系统是否真正解决了问题**这件事必须有人类参与。评估本身需要人类定义标准、审查边界、判断可接受性。

**OpenAI** 的官方建议直指生产环境的核心风险：

> "生产环境锁定模型版本号，建立衡量提示词性能的eval。"

模型provider随时可能更新模型行为。没有eval，你甚至不知道系统在悄悄退化。

**LangChain团队** 用实际数据证明了评估驱动迭代的威力：

> 仅改治具不改模型，Terminal Bench排名从Top 30升至Top 5（52.8% → 66.5%）。

这个案例说明：**有评估闭环的系统化迭代，比换一个更强的模型更有效。** 因为你知道哪里差、知道改了什么、知道改善了多少。
