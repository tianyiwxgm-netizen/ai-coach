# 治具工程：让AI Agent可靠运行的约束系统

> **核心洞见：治具决定成败，模型是大宗商品。围绕AI Agent构建的脚手架、约束和反馈循环——而非模型本身——才是系统可靠性和工程效率的真正来源。**

---

## 一、为什么有效：从提示词到治具的必然演进

### 1.1 方法论演进链

```
提示词工程 (2023-2024)
 "怎么写出好的指令？"
    │
    │ 局限：单条指令无法解决复杂任务
    ▼
上下文工程 (2025)
 "怎么设计模型的整体信息环境？"
    │
    │ 局限：信息环境好了，但Agent多步执行仍不可靠
    ▼
治具工程 (2026.2)
 "怎么构建让Agent可靠运行的完整约束系统？"
```

每一层的诞生都是对前一层局限性的直接回应。提示词工程解决了"单次调用怎么写"，上下文工程解决了"模型该看到什么"，但当AI从"回答问题的工具"进化为"自主执行多步任务的Agent"时，一个全新的问题类别出现了——这些问题不在"模型该看到什么"的范畴内。

### 1.2 什么是治具

**治具（Harness）** 这个词来自制造业：夹具、模具、测试台架——不是产品本身，而是确保产品被正确制造的整套环境。

Mitchell Hashimoto在2026年2月5日的博客中首次系统阐述了这一概念：**治具是围绕AI Agent的完整环境——脚手架（scaffolding）、约束（constraints）、反馈循环（feedback loops）。** 治具不是Agent本身，是管理Agent运行全生命周期的软件系统。

6天后，2月11日，OpenAI发布了内部Codex团队的治具工程报告，用5个月的纯AI编码实验验证了同一结论：系统可靠性的飞跃来自治具的迭代改善，而非模型升级。

**核心区分：**
- Agent = 由LLM驱动的自主执行体（决定做什么、怎么做）
- 治具 = 包裹Agent的运行时系统（决定Agent能做什么、不能做什么、做错了怎么办）

类比软件工程：Agent是你的应用程序，治具是操作系统 + CI/CD + 监控 + 安全边界。你不会让一个应用程序裸跑在没有操作系统的硬件上，同样不应该让Agent裸跑在没有治具的环境中。

### 1.3 为什么上下文工程不够

上下文工程关注的是：给模型正确的信息，让它做出正确的决策。这在单次调用中效果显著。但Agent多步执行时，一类全新的问题出现了：

| 问题类别 | 表现 | 上下文工程能解决吗？ |
|---------|------|-------------------|
| 忽略团队规范 | Agent生成的代码不遵循项目的命名约定、目录结构 | 部分——可以把规范放入上下文，但Agent经常在长任务中"遗忘" |
| 违反架构约束 | Agent引入了错误的依赖关系，破坏了模块边界 | 否——这需要确定性检查，不能靠LLM"记住" |
| 代码质量熵增 | 随着Agent持续产出代码，技术债务快速累积 | 否——这需要周期性的清理和重构机制 |
| 安全漏洞 | Agent引入硬编码密钥、SQL注入、不安全的依赖 | 否——这需要自动化扫描和阻断 |
| 错误级联 | 一个步骤的错误传播到后续所有步骤 | 否——这需要检查点、回滚和错误恢复机制 |

**关键洞见：这不是"模型该看到什么"的问题，而是"系统该阻止什么、度量什么、修复什么"的问题。**

此外，上下文本身是稀缺资源。Anthropic的研究表明，LLM推理性能在约3000 tokens时开始下降。如果你把所有团队规范、架构约束、安全规则都塞进系统提示，巨大的指令文件会挤占实际任务的上下文空间，反而降低Agent的执行质量。治具工程的精妙之处在于：**把约束从上下文中剥离出来，交给确定性系统执行。**

## 二、怎么用：治具的核心组件

### 2.1 Martin Fowler的四层模型

Martin Fowler在其对治具工程的系统分析中，将治具分解为四个协作层。这是目前最清晰的架构框架：

```
┌─────────────────────────────────────────────────┐
│           上下文工程层                             │
│  知识库 + 动态上下文（可观测性数据、运行时状态）    │
│  AGENTS.md作为目录（不是百科全书）                 │
│  docs/作为系统记录                                │
├─────────────────────────────────────────────────┤
│           架构约束层                               │
│  LLM-based Agent检查（语义级规范理解）             │
│  + 确定性自定义linter（类型检查、import规则等）     │
│  混合方法：确定性规则处理能确定的，LLM处理模糊的    │
├─────────────────────────────────────────────────┤
│           "垃圾回收"层                            │
│  周期性运行的后台Agent                             │
│  查找代码库中的不一致性                            │
│  对抗持续的熵增                                    │
├─────────────────────────────────────────────────┤
│           安全与治理层                             │
│  预检查（lint / typecheck / secret scan）          │
│  PR级审查套件                                     │
│  金丝雀部署 + 监控                                │
└─────────────────────────────────────────────────┘
```

**每一层都有其不可替代的职责：**

**上下文工程层** 是治具的信息基础。但关键设计原则是：AGENTS.md应该是目录而非百科全书。它告诉Agent"去哪里找信息"，而不是把所有信息都塞给Agent。具体的规范、约定、架构决策记录在docs/目录中，Agent按需检索。

```
# AGENTS.md（正确做法：目录）
## 架构
见 docs/architecture.md

## 编码规范
见 docs/coding-standards.md

## 测试约定
见 docs/testing-guide.md

# 而不是把所有内容都写在AGENTS.md里
```

**架构约束层** 是治具最核心的创新。Martin Fowler特别指出：**这一层混合了确定性方法和LLM-based方法。** 能用确定性规则检查的（import限制、类型检查、命名约定），用自定义linter；需要语义理解的（"这个函数是否违反了单一职责原则"），用LLM-based Agent检查。两者互补，缺一不可。

**"垃圾回收"层** 是应对Agent编码熵增的关键机制。即使每一次Agent产出都通过了约束检查，随着时间推移，代码库仍然会出现不一致性——重复的抽象、漂移的命名风格、冗余的依赖。就像编程语言的垃圾回收器一样，需要周期性运行的后台Agent来发现和修复这些问题。

### 2.2 OpenAI Codex团队的实验数据

OpenAI Codex团队的实验是治具工程最有力的实证：

**规模：** 5个月纯AI编码，团队从3人扩展到7人，产出约1500个PR、约100万行代码，零手写代码。

**初期困境：** 生产力极低。原因不是模型能力不足，而是：
- 环境设置不完善——Agent不知道怎么初始化项目环境
- 工具集成弱——Agent缺少必要的开发工具
- 错误恢复差——Agent在遇到错误时无法有效自救

**治具改善过程：** 团队不是在调优提示词，而是在构建和迭代治具系统。每一轮治具改善都带来可度量的生产力提升。

**结果：** 治具成熟后，效率提升约10倍。

**关键治具组件：**

```
初始化器（Initializer）
  Agent启动时自动执行：
  - 克隆仓库并安装依赖
  - 运行一次完整测试确认环境健康
  - 加载项目上下文（AGENTS.md → 按需读取docs/）
  - 配置工具链（linter、formatter、类型检查器）

编码Agent模式（Coding Agent Pattern）
  Agent执行循环：
  - 阅读任务描述和相关代码
  - 生成实现代码
  - 运行linter和类型检查（确定性约束）
  - 运行测试套件
  - 如果失败：分析错误 → 修复 → 重新检查（反馈循环）
  - 通过所有检查后提交PR
```

### 2.3 流程工程：设计调用序列

治具工程不仅关注单次Agent执行的约束，还关注**多次LLM调用之间的编排逻辑**——这就是流程工程（Flow Engineering）。

流程工程的核心问题不是"如何优化单次LLM调用的内容"，而是：

```
什么时候该调用LLM？
  → 需要语义理解和判断的环节
  → 不是所有步骤都需要LLM——能用确定性逻辑的用确定性逻辑

用什么数据调用？
  → JIT（Just-In-Time）上下文注入
  → 每次调用只给这一步所需的上下文，不是全量上下文

出错了怎么办？
  → 定义每个节点的失败策略：重试、回退、跳过、人工介入
  → 设置超时和资源限制
  → 保存检查点，支持断点续跑
```

**流程工程的设计产物是一张有向图：**

```
任务解析 → 代码检索 → [LLM] 方案设计 → [LLM] 代码生成
                                                ↓
                                         确定性检查 ←───┐
                                         (lint/type)    │
                                              ↓         │
                                         测试执行        │
                                              ↓         │
                                         通过？──否──→ [LLM] 错误修复
                                              │
                                             是
                                              ↓
                                         安全扫描 → PR创建
```

图中的方括号`[LLM]`标注了需要LLM的节点。其他节点是确定性执行。这种混合设计是治具工程的核心思想。

## 三、场景实操指南

### 3.1 场景一：AI辅助开发（个人/小团队）

**问题：** 你用Claude Code或Cursor编写代码，但输出质量不稳定。

**治具方案：**

```
第一层：上下文治具
  ├── CLAUDE.md / AGENTS.md：项目架构概述（目录式）
  ├── docs/：编码规范、架构决策记录、API文档
  └── .cursorrules / .clinerules：IDE级别的行为约束

第二层：约束治具
  ├── pre-commit hooks：lint + format + type-check
  ├── 自定义规则：import限制、禁用列表、架构边界检查
  └── 测试套件：单元测试 + 集成测试作为验收标准

第三层：反馈治具
  ├── CI/CD pipeline：PR自动检查
  ├── 代码审查检查清单（人工 + AI辅助）
  └── 定期代码健康度量（复杂度、覆盖率、依赖新鲜度）
```

**关键原则：** 约束越多不是越好——而是约束越精准越好。无效的约束会消耗Agent的注意力预算，和无效的上下文一样有害。

### 3.2 场景二：生产级Agent系统

**问题：** 你构建了一个自主执行任务的Agent系统（如自动化测试、数据处理管线、内容生产流水线），但可靠性不足。

**治具方案：**

```
初始化治具
  ├── 环境验证：启动前检查所有依赖和权限
  ├── 状态恢复：从上次中断点继续而非从头开始
  └── 工具注册：Agent可用工具的显式声明和权限控制

运行时治具
  ├── 令牌预算管理：防止无限循环消耗资源
  ├── 操作审计日志：每个Agent动作都有记录
  ├── 沙箱执行：Agent生成的代码在隔离环境运行
  └── 人工审批门：高风险操作需要人工确认

后处理治具
  ├── 输出验证：Schema检查 + 语义一致性检查
  ├── 回归测试：确认新输出不破坏现有功能
  └── 指标收集：延迟、成功率、资源消耗
```

### 3.3 场景三：大规模团队AI采用

**问题：** 团队50+人，多个Agent系统并行开发，需要统一的治理框架。

**治具方案：**

```
平台级治具
  ├── 统一Agent运行时：标准化的初始化、监控、日志
  ├── 共享知识库：团队级的docs/、规范、最佳实践
  ├── 安全基线：所有Agent必须通过的安全检查清单
  └── 成本监控：Token消耗和API调用的可见性

团队级治具
  ├── 领域特定约束：每个团队根据自己的技术栈定制
  ├── 评估体系：团队自定义的质量基准
  └── 渐进式自主度：新Agent从低自主度开始，验证后逐步提升

个人级治具
  ├── IDE配置：统一的开发环境设置
  ├── 提示词库：经过验证的高效提示词模板
  └── 本地预检查：提交前的快速验证
```

### 4.1 AGENTS.md目录模板

```markdown
# AGENTS.md

## 项目概览
[1-2句话描述项目是什么、做什么]

## 技术栈
- 语言/框架：[具体版本]
- 数据库：[类型]
- 部署：[环境]

## 快速命令
- 启动：`[命令]`
- 测试：`[命令]`
- 构建：`[命令]`

## 规范（按需查阅docs/）
- 编码规范：docs/coding-standards.md
- 架构决策：docs/architecture-decisions/
- API约定：docs/api-conventions.md
- 测试策略：docs/testing-strategy.md

## 关键约束（必须遵守）
- [ ] 不要修改 [核心文件/目录]，除非被明确要求
- [ ] 所有公共API必须有类型注解
- [ ] 新依赖必须说明理由并更新 docs/dependencies.md
- [ ] 安全敏感操作必须记录审计日志

## 当前活跃任务
见 issues/ 或项目管理工具
```

**设计要点：** 控制在200 tokens以内。所有详细内容通过链接指向docs/，Agent按需读取。

### 4.2 治具检查清单模板

```markdown
# Agent治具检查清单

## 上下文层 ✓/✗
- [ ] AGENTS.md存在且保持目录式（<200 tokens）
- [ ] docs/包含所有必要规范
- [ ] 动态上下文有JIT检索机制
- [ ] 上下文中无过时或冗余信息

## 约束层 ✓/✗
- [ ] 有pre-commit hooks（lint + format + typecheck）
- [ ] 有自定义架构约束规则
- [ ] 确定性检查和LLM检查各司其职
- [ ] Agent不能绕过约束

## 反馈层 ✓/✗
- [ ] 错误信息能传回Agent
- [ ] 有自动重试和错误恢复机制
- [ ] 有超时和资源限制
- [ ] 有人工介入触发条件

## 垃圾回收层 ✓/✗
- [ ] 有周期性代码健康检查
- [ ] 有不一致性检测机制
- [ ] 有技术债务度量

## 安全层 ✓/✗
- [ ] 有密钥扫描（secret scanning）
- [ ] 有依赖漏洞检查
- [ ] 高风险操作需人工审批
- [ ] Agent运行在沙箱中
```

### 4.3 治具架构模板（以AI编码Agent为例）

```python
class CodingHarness:
    """AI编码Agent治具的概念架构"""

    def initialize(self, task):
        """初始化治具——Agent启动前执行"""
        self.clone_and_setup_environment()
        self.run_health_check()  # 确认环境可用
        self.load_context(task)   # JIT加载相关上下文
        self.register_tools()     # 注册Agent可用工具
        self.set_resource_limits()  # 设置token和时间预算

    def run_agent_loop(self, task):
        """Agent执行循环——核心治具逻辑"""
        for attempt in range(self.max_attempts):
            # 1. Agent生成代码（LLM调用）
            code = self.agent.generate(task, self.context)

            # 2. 确定性约束检查（不用LLM）
            lint_result = self.run_linter(code)
            type_result = self.run_type_checker(code)
            if not (lint_result.ok and type_result.ok):
                self.context.add_feedback(lint_result, type_result)
                continue  # 反馈给Agent重试

            # 3. 测试验证
            test_result = self.run_tests()
            if not test_result.ok:
                self.context.add_feedback(test_result)
                continue

            # 4. 安全扫描
            security_result = self.run_security_scan(code)
            if security_result.has_critical:
                self.alert_human(security_result)
                return  # 阻断，等待人工介入

            # 5. 全部通过
            return self.create_pr(code)

        # 超过最大重试次数
        self.escalate_to_human(task)

    def garbage_collect(self):
        """周期性运行——对抗代码库熵增"""
        issues = self.scan_for_inconsistencies()
        for issue in issues:
            if issue.auto_fixable:
                self.create_fix_pr(issue)
            else:
                self.create_issue(issue)
```

## 五、代理工程：人在治具中的位置

### 5.1 Karpathy的代理工程

Andrej Karpathy提出了一个关键概念：**代理工程（Agentic Engineering）**——AI做实现，人拥有架构、质量和正确性。

这不是"让AI随便写代码然后祈祷它能工作"。代理工程有严格的工作流：

```
先规划      人类制定架构、定义接口、设计数据模型
    ↓
指导审查    AI生成代码，人类审查关键决策和品味问题
    ↓
重测试      测试覆盖率不降低，关键路径有端到端测试
    ↓
拥有代码库  人类理解每一行代码的意图，即使不是人写的
```

### 5.2 代理工程 vs Vibe Coding

| 维度 | Vibe Coding | 代理工程 |
|------|------------|---------|
| 计划 | "先写起来再说" | 先设计架构、接口、约束 |
| 审查 | "看起来能跑就行" | 审查每个PR的设计决策 |
| 测试 | "手动试几下" | 自动化测试套件作为验收标准 |
| 所有权 | "AI写的我也不太懂" | "虽然AI写的但我完全理解" |
| 适用场景 | 原型、实验、一次性脚本 | 生产系统、团队项目、长期维护 |

**核心洞见：好的工程实践在AI辅助开发中更重要而非更不重要。** 当AI每小时能产出以前一天才能完成的代码量时，如果没有约束和审查机制，代码质量的下降速度也会加倍。

这也解释了为什么**高级工程师从AI中受益更大**：他们理解系统设计、安全边界、性能瓶颈和维护成本。他们能为Agent设计出更好的约束，能更快地发现Agent产出中的问题，能在更高的抽象层次上指导AI工作。治具的质量直接取决于构建治具的人对软件工程的理解深度。

### 5.3 "AI Slop"：治具缺失的代价

"AI slop"是2025-2026年社区形成的一个重要概念：看起来合理但实际上有问题的AI生成代码。其典型特征：

- **缺少错误处理：** 只处理了happy path，异常路径一片空白
- **有安全漏洞：** 硬编码密钥、未验证输入、不安全的默认配置
- **破坏架构：** 引入不合理的依赖、违反模块边界、创建God Object
- **过度工程：** 为简单问题引入复杂的设计模式和抽象层
- **表面正确：** 通过了基本测试但有边界条件bug

**AI slop不是模型的问题，是治具缺失的问题。** 有完善治具的团队不会遇到AI slop，因为每一类问题都有对应的检查机制在Agent提交代码之前拦截它。

## 六、安全与治理

### 6.1 为什么安全在治具工程中是一等公民

Agent可以大规模引入漏洞。人类开发者手动写代码时，引入漏洞的速度受限于编码速度。Agent没有这个限制——它能在一小时内产出数千行代码，每一行都可能包含安全问题。

更危险的是：AI生成的代码看起来很"专业"。它有注释、有文档字符串、有类型注解——但内层逻辑可能有SQL注入、路径遍历或权限绕过。审查者很容易被表面的"专业感"欺骗而放松警惕。

**安全必须内建于治具而非后装。** 后装意味着依赖人记得去做安全审查——而在AI大量产出代码的场景下，这不可行。

### 6.2 安全治具的三道防线

```
第一道：预检查（Agent提交代码前）
  ├── lint/typecheck：捕获基础代码质量问题
  ├── secret scan：检测硬编码的密钥、token、密码
  ├── 依赖检查：已知漏洞的依赖库
  └── 架构规则：禁止的import、API调用、模式

第二道：PR审查套件（代码提交后、合并前）
  ├── SAST（静态应用安全测试）
  ├── 覆盖率检查：安全敏感路径必须有测试
  ├── LLM-based安全审查：检测复杂的逻辑漏洞
  └── 人工审查：高风险变更必须有安全工程师签字

第三道：金丝雀部署（代码合并后）
  ├── 渐进式发布：先1%流量验证
  ├── 异常检测：错误率、延迟、资源消耗的突变
  ├── 自动回滚：异常指标触发自动回滚
  └── 安全监控：运行时威胁检测
```

## 七、刻意练习：5周训练方案

### 第1周：上下文治具

**目标：** 为你的主项目建立合格的上下文层。

**练习：**
1. 审计你现有的AGENTS.md或CLAUDE.md——它是目录还是百科全书？如果超过500 tokens，精简它。
2. 把详细内容迁移到docs/，每个主题一个文件。
3. 让Agent基于你的上下文层完成3个不同任务，观察它是否能找到所需信息。
4. 记录Agent失败的点，补充缺失的文档。

### 第2周：约束治具

**目标：** 建立第一批自动化约束。

**练习：**
1. 配置pre-commit hooks：lint + format + typecheck。
2. 编写至少3条自定义linter规则（如：禁止直接import某个内部模块、强制使用特定的日志函数、检测硬编码URL）。
3. 让Agent故意违反这些规则，验证约束是否生效。
4. 思考哪些约束需要LLM-based检查而非确定性规则。

### 第3周：反馈循环

**目标：** 让Agent的错误能自动回流。

**练习：**
1. 构建一个最小的Agent执行循环：生成代码 → 运行测试 → 失败时把错误信息反馈给Agent → 重试。
2. 设置最大重试次数和超时时间。
3. 度量：第一次生成通过率 vs 带反馈循环的最终通过率。
4. 记录Agent无法自修复的错误类型，思考需要哪些额外约束。

### 第4周：垃圾回收

**目标：** 建立对抗熵增的机制。

**练习：**
1. 用Agent扫描代码库，查找不一致性（命名风格不统一、重复的抽象、未使用的import等）。
2. 对每个发现的问题分类：自动修复 vs 需要人工决策。
3. 为自动修复的问题创建定期运行的脚本。
4. 建立代码健康度量dashboard：复杂度趋势、覆盖率趋势、依赖新鲜度。

### 第5周：端到端整合

**目标：** 把前4周的组件整合为完整治具。

**练习：**
1. 完成一个完整流程：Agent接收任务 → 初始化 → 执行 → 约束检查 → 测试 → 安全扫描 → 提交PR。
2. 让5个不同的任务走完全流程，记录成功率和失败点。
3. 对比有治具和无治具的Agent产出质量（用你在第2周定义的规则度量）。
4. 撰写你的治具文档：什么有效、什么无效、下一步改进方向。

## 八、反模式：这样做会失败

### 反模式1：百科全书式AGENTS.md

**表现：** 把所有编码规范、架构文档、API说明都塞进一个巨大的AGENTS.md文件。

**为什么失败：** 上下文是稀缺资源。3000+ tokens的指令文件会挤占任务上下文空间，导致Agent对实际任务的理解和执行质量下降。LLM的注意力被稀释——就像给一个员工同时布置了50项规范要求和1个具体任务，他大概率记不住所有规范。

**正确做法：** AGENTS.md是目录（<200 tokens），详细内容在docs/中，Agent按需检索。

### 反模式2：全LLM约束，零确定性规则

**表现：** 所有代码质量检查都用LLM做——"让AI审查AI的代码"。

**为什么失败：** LLM是概率性的。"检查import是否符合规范"这种有明确规则的检查，用LLM做意味着每次结果可能不同。更重要的是，LLM检查消耗Token和时间，而linter在毫秒内完成且结果100%确定。

**正确做法：** Martin Fowler的混合原则——能用确定性规则的用确定性规则，只有需要语义理解的检查才用LLM。

### 反模式3：无限重试，无退出条件

**表现：** Agent失败后无限重试，直到成功或耗尽预算。

**为什么失败：** 有些错误Agent本身无法修复——架构层面的设计缺陷、模糊的需求、不存在的API。无限重试不仅浪费资源，还可能让Agent在错误方向上越走越远。

**正确做法：** 设置明确的最大重试次数（通常3-5次）、超时时间和退出条件。超限后升级到人工介入。

### 反模式4：先构建Agent，后补治具

**表现：** 先让Agent跑起来，"等出了问题再加约束"。

**为什么失败：** 没有治具的Agent会快速产生大量低质量代码，形成技术债务。后期补治具时，你不仅要建约束系统，还要清理已经积累的debt。OpenAI Codex团队的经验证明：**治具应该先于Agent或与Agent同时构建。**

**正确做法：** 先搭建最小可行治具（基本的lint + test + 安全扫描），然后在这个治具内启动Agent，逐步迭代两者。

### 反模式5：忽略安全，信任Agent的"专业感"

**表现：** AI生成的代码看起来很专业（有类型注解、文档字符串、注释），审查时因此放松警惕。

**为什么失败：** AI生成代码的"专业外观"和"实际安全性"是独立的维度。LLM优化的是"看起来像好代码"（因为训练数据中好代码通常有注释和类型注解），但它不会主动考虑安全边界——除非约束系统要求它这么做。

**正确做法：** 安全扫描必须是自动化的、不可绕过的。不管代码看起来多"专业"，都必须通过secret scan、SAST和依赖检查。

## 九、专家怎么说

### Mitchell Hashimoto（HashiCorp创始人，治具工程先驱）

> "治具决定成败，模型是大宗商品。"

2026年2月5日，Hashimoto发表了关于治具工程的开创性博客。核心论点：经过5个月纯AI编码实验，他发现**初期的低生产力完全是治具问题，而非模型问题**。当治具（环境设置、工具集成、错误恢复、约束检查）逐步完善后，同样的模型产出了10倍的效率。约100万行代码，零手写——但这个结果完全依赖于精心构建的治具系统。

### Martin Fowler（ThoughtWorks首席科学家）

> "治具组件混合确定性和LLM-based方法。"

Fowler在其分析文章中系统化了治具的组件架构。他特别强调了**混合方法**的重要性：不是所有约束都应该用LLM实现，也不是所有约束都能用确定性规则覆盖。最有效的治具在每个检查点选择最合适的方法——linter处理能确定的，LLM处理需要语义理解的。

### LangChain团队

> 仅改治具（不改模型），Terminal Bench排名从Top 30跳到Top 5。

这是治具工程最有说服力的量化证据之一。LangChain团队在Terminal Bench 2.0基准测试中，Agent得分从52.8%提升到66.5%。**模型完全没有变化**，所有性能提升纯粹来自治具的优化——更好的上下文管理、更精确的工具描述、更有效的错误恢复机制。

### Anthropic

> Claude Agent SDK是"通用Agent治具"。

Anthropic将其Agent SDK定位为通用治具框架，提供了初始化、工具管理、安全沙箱、反馈循环等治具基础设施。在其官方文档中，Anthropic明确指出：**有效的长时间运行Agent的关键不在于更好的提示词，而在于更好的治具**——包括清晰的工具定义、健壮的错误处理、适当的人工介入点。

### Andrej Karpathy

> "好的工程实践在AI辅助开发中更重要而非更不重要。"

Karpathy通过代理工程概念框架，将治具思想扩展到人机协作层面。他的核心论点：当AI的执行速度远超人类时，工程约束（测试、审查、架构规范）不是可以放松的"累赘"，而是必须加强的"安全带"。高级工程师受益更大，因为他们能设计出更好的约束系统。

## 十一、总结：治具工程的核心原则

```
1. 治具先于Agent或与Agent同步构建，永远不要"后补"
2. 约束混合确定性方法和LLM方法——各取所长
3. AGENTS.md是目录不是百科全书——保护上下文预算
4. 安全内建于治具——不可绕过的自动化检查
5. 垃圾回收对抗熵增——周期性清理不一致性
6. 人类拥有架构和品味——AI拥有执行速度
7. 度量一切——没有评估就没有改进
```

**治具工程的终极目标不是让Agent更聪明，而是让Agent在一个精心设计的约束系统中可靠地运行。** 就像高速公路的护栏不会让车跑得更快，但能确保车安全到达目的地。模型能力会持续提升，但再强的模型裸跑也不如中等模型加好的治具。这就是为什么投资治具——而非追逐最新模型——才是真正的工程护城河。
