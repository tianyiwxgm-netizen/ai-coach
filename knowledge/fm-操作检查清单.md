# 操作检查清单

> 每个清单都是实战提炼，按顺序逐项检查。

---

## 1. 写提示词前的检查清单

开始写提示词之前，先回答这5个问题：

- [ ] **目标明确？** 我能用一句话说清楚想要什么输出吗？
- [ ] **有样本吗？** 我手头有没有"好输出"的例子可以作为少样本示例？
- [ ] **上下文够吗？** AI需要知道哪些背景信息才能给出好答案？（领域知识、约束条件、受众）
- [ ] **输出格式定了吗？** JSON / Markdown / 表格 / 代码？要明确指定。
- [ ] **评估标准有吗？** 我怎么判断输出是好是坏？能写成评分标准吗？

> 如果5项都答不出来，先别急着写提示词——先想清楚需求。

---

## 2. Agent开发检查清单

从零开发一个AI Agent时逐项确认：

- [ ] **任务定义** — Agent要完成的任务是否可以被清晰定义？模糊任务不适合Agent。
- [ ] **工具清单** — 列出Agent需要的所有工具，每个工具的输入输出是否明确？
- [ ] **系统提示词** — 是否写了清晰的角色定义、行为边界、工具使用规则？
- [ ] **状态管理** — Agent需要记住什么？对话历史如何管理？上下文窗口会不会溢出？
- [ ] **错误处理** — 工具调用失败怎么办？LLM幻觉怎么办？死循环怎么办？
- [ ] **护栏设置** — 最大迭代次数、超时限制、输入/输出校验是否都设了？
- [ ] **人机交互点** — 哪些决策需要人类确认？确认机制怎么实现？
- [ ] **评估方案** — 用什么测试集？成功标准是什么？如何回归测试？

> 关键原则：先用确定性工作流解决，解决不了再上Agent。

---

## 3. RAG系统部署检查清单

搭建RAG（检索增强生成）系统时逐项确认：

- [ ] **数据预处理** — 文档清洗了吗？切分策略合理吗？chunk大小测试过了吗？（建议从512 token开始）
- [ ] **Embedding选型** — 选了什么Embedding模型？和查询语言匹配吗？维度/性能平衡了吗？
- [ ] **检索策略** — 纯向量检索够用吗？需要混合检索（向量+关键词）吗？需要重排序吗？
- [ ] **上下文组装** — 检索到的chunk怎么拼进提示词？顺序、数量、去重都考虑了吗？
- [ ] **引用追溯** — 能追溯到源文档吗？用户能验证答案来源吗？
- [ ] **效果评估** — 检索准确率、回答质量怎么量化？用RAGAS还是人工评估？
- [ ] **持续更新** — 新文档怎么入库？旧文档怎么失效？增量更新还是全量重建？

> 常见陷阱：chunk切太小丢上下文，切太大引入噪声。从512 token开始，根据评估结果调整。

## 4. AI产品上线前检查清单

AI功能推向用户前的终极检查：

- [ ] **功能正确性** — 核心用例全部测试通过？边界情况覆盖了吗？
- [ ] **输出质量** — 在真实数据上评估过了吗？人工审核了一批输出吗？
- [ ] **延迟性能** — P50/P95延迟可接受吗？流式输出配置了吗？
- [ ] **成本预算** — 单次调用成本算了吗？高峰期成本估了吗？有成本报警吗？
- [ ] **错误处理** — API超时/限流/报错的降级方案有吗？用户看到的错误信息友好吗？
- [ ] **安全防护** — 提示注入防了吗？敏感信息泄露防了吗？输出过滤配了吗？
- [ ] **可观测性** — 日志记全了吗？能追踪每次LLM调用的输入输出吗？（Langfuse等）
- [ ] **用户反馈** — 用户能对AI输出点赞/踩吗？反馈数据能回收用于改进吗？
- [ ] **回滚方案** — 出问题能快速回滚到非AI方案吗？有feature flag吗？
- [ ] **合规审查** — 数据使用合规吗？用户知道在和AI交互吗？保留审计日志了吗？

> 第一版不需要全部完美，但至少要知道每一项的状态。

## 5. 提示词调优迭代检查清单

提示词效果不好时的系统化改进流程：

- [ ] **收集失败样本** — 至少收集5-10个bad case，分类失败原因（格式错？内容错？遗漏？幻觉？）
- [ ] **检查上下文** — 该给的背景信息都给了吗？是不是缺了关键约束条件？
- [ ] **添加示例** — 加1-3个少样本示例，特别是边界情况的示例
- [ ] **分步骤拆解** — 一个大提示拆成多步（先分析→再生成→再检查）
- [ ] **添加自检环节** — 让模型在输出前先自检："检查你的答案是否满足所有要求"
- [ ] **A/B测试** — 用同一批测试集对比改进前后的效果，量化提升

> 迭代原则：每次只改一个变量，记录每次修改和效果变化。

---

## 6. 代码审查（AI生成代码）检查清单

审查AI写的代码时重点检查：

- [ ] **逻辑正确性** — 不要相信"看起来对"，跑一遍测试。AI最擅长写看起来正确但细节有误的代码。
- [ ] **边界条件** — 空输入、null值、超大数据、并发场景都处理了吗？
- [ ] **错误处理** — 异常捕获合理吗？不是所有异常都catch掉就行。
- [ ] **安全性** — SQL注入、XSS、硬编码密钥、不安全的依赖？
- [ ] **性能** — 有没有不必要的循环、重复查询、内存泄漏？N+1问题？
- [ ] **可维护性** — 命名清晰吗？注释有意义吗？不要保留AI的废话注释。
- [ ] **依赖合理** — 引入了不必要的依赖吗？版本兼容吗？许可证合规吗？
- [ ] **测试覆盖** — 有测试吗？测试覆盖了核心逻辑和边界情况吗？

> 核心心态：AI写的代码 = 实习生写的代码，质量可能不错但必须review。

---

## 7. 安全审查检查清单

AI系统的安全专项检查：

- [ ] **提示注入** — 用户输入会直接拼进提示词吗？做了分隔和转义吗？测试过对抗样本吗？
- [ ] **数据泄露** — 系统提示词能被套出来吗？训练数据/内部信息会泄露吗？
- [ ] **权限边界** — Agent能访问的工具/数据是否最小化？有没有越权操作的可能？
- [ ] **输出过滤** — 有没有对AI输出做安全过滤？（有害内容、个人信息、敏感数据）
- [ ] **速率限制** — API调用有速率限制吗？能防止滥用和拒绝服务吗？
- [ ] **审计日志** — 所有AI交互都有日志吗？能事后审查异常行为吗？

> 安全不是一次性的，需要持续监控和定期审查。
