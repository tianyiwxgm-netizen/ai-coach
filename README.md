## 30 秒了解核心价值

```
你：教练，我要修一个优惠券不生效的 Bug

教练：我来帮你梳理。几个问题：
      1. 能稳定复现吗？
      2. 相关代码在哪？
      3. Console 有报错吗？

你：每次都能复现。代码在 module/oms/，Console 没报错

教练：信息够了。推荐流程：
      1. 先写复现测试 → 确认 FAIL（证明 Bug 存在）
      2. 定位根因
      3. 修复 → 测试 PASS
      4. 提交

      这是示范提示词，复制到新会话执行：
      ┌─────────────────────────────────────┐
      │ 我需要修复优惠券不生效的 Bug...       │
      │ 请用 /systematic-debugging 定位，    │
      │ 再用 /test-driven-development 修复。 │
      └─────────────────────────────────────┘
```

**教练的产出不是代码，而是：诊断结论 + 工作流推荐 + 可直接执行的提示词。**

## 5 种专业角色

首次使用选择角色，之后自动记住。不同角色有完全不同的引导流程和评估维度。

| 角色 | 适合谁 | 教练怎么帮你 |
|------|--------|-------------|
| **一人全栈** | 独立开发者、创业者 | 全类型任务诊断（Bug/新功能/重构/紧急故障），从需求到交付全链路引导 |
| **产品经理** | PM、需求分析师 | 历史资产盘点 → 业务分析 → AI 可执行 PRD → 可演示原型 → 交付检查 |
| **后端开发** | 服务端工程师 | 技术方案设计 → 6 阶段 TDD 实施 → 后台页面 → 自测交付 |
| **测试工程师** | QA、SDET | 测试策略 → 三层用例设计 → 自动化执行 → 标准测试报告 |
| **前端开发** | Web/iOS/Android/小程序 | 技术选型 → 组件拆分 → 多端适配 → e2e 验证 |

## 安装

**前置条件：** 已安装 [Claude Code CLI](https://docs.anthropic.com/en/docs/claude-code)

```bash
# 1. 克隆仓库
git clone https://github.com/tianyiwxgm-netizen/ai-coach.git ~/.claude/skills/ai-coach

# 2. 运行安装脚本
bash ~/.claude/skills/ai-coach/install.sh
```

**验证：** 重启 Claude Code，输入 `教练` 或 `/ai-coach`，看到角色选择菜单即安装成功。

**更新：**
```bash
cd ~/.claude/skills/ai-coach && git pull
```

**卸载：**
```bash
bash -c 'for s in fullstack product backend testing frontend init evaluate enterprise; do rm -f ~/.claude/skills/ai-coach-$s; done'
rm -rf ~/.claude/skills/ai-coach
```

## 使用场景

### 场景 1：修 Bug — 不知道从哪下手

```
> 教练，用户下单时选了优惠券，但结算价格没变

教练会：
1. 追问复现条件、报错信息、代码位置
2. 诊断出是"优惠金额计算逻辑问题"
3. 给出 Bug 修复铁律：写复现测试 → 定位根因 → 修复 → 验证
4. 产出一段可直接粘贴到新会话的提示词
```

### 场景 2：开发新功能 — 刚拿到需求

```
> 教练，我要做批量导入商品功能

教练会：
1. 追问格式、数据量、现有功能
2. 推荐工作流：brainstorming → writing-plans → TDD 实施
3. 产出结构化提示词，包含所有上下文
```

### 场景 3：写 PRD — 产品经理角色

```
> 教练，我要写供应商管理系统 PRD

教练会：
1. 先盘点 6 项历史资产（现有功能/旧 PRD/表结构/接口/运营数据/用户反馈）
2. 引导业务分析 → 竞品分析 → 能力差距矩阵
3. 产出 10 章标准 PRD（达到"不问我就能开始写代码"的水准）
```

### 场景 4：后端开发 — 含后台管理页面

```
> 教练，我要开发供应商管理功能，有列表页和编辑弹窗

教练会：
1. 收集 PRD、模块路径、页面字段、操作按钮
2. 规划 6 阶段 TDD：数据层 → 服务层 → 接口层 → 页面层 → e2e → 交付
3. 每阶段产出独立的提示词，在新会话中执行
```

### 场景 5：自动化测试规划

```
> 教练，货物管理开发完了，帮我规划测试

教练会：
1. 收集 PRD、技术方案、代码路径
2. 设计测试金字塔：单元(60%) + 接口(30%) + e2e(10%)
3. 三层用例设计表 → 自动化执行方案 → 测试报告模板
```

### 场景 6：前端移动端开发

```
> 教练，做一个 iOS 巡检 App 的扫码页面

教练会：
1. 追问平台、技术栈、设计稿、兼容要求
2. 技术选型引导（Swift/Flutter/React Native/小程序…）
3. 6 阶段实施：基础 → 组件 → 页面 → 联调 → 适配 → e2e
```

### 场景 7：周五复盘 — 量化你的 AI 使用水平

```
> 复盘

教练会分析你的 Claude Code 会话记录，按角色专属 8 维评分：

  测试上下文完整度   ★★★★☆  72/100  ✓ 做得不错
  用例覆盖率         ★★☆☆☆  40/100  ⚠ 最需改进
  自动化比例         ★★☆☆☆  35/100  ⚠ 最需改进

  TOP 2 改进建议：
  1. 自动化比例 — 5 个任务只有 1 个写了自动化，建议...
  2. 用例覆盖率 — 只覆盖正常场景，缺少异常和边界...
```

### 场景 8：紧急线上故障

```
> 教练，线上挂了！登录报 500

教练识别到紧急情况，跳过常规流程：
1. 快速定位三连问（日志/范围/最近部署）
2. 给出止血方案（回滚 vs 定点修复）
3. 事后必须补：测试 + 复盘
```

## 工作原理

```
                        ┌──────────────┐
                        │   AI Coach   │
                        │   角色路由    │
                        └──────┬───────┘
               ┌───────┬───────┼───────┬───────┐
               v       v       v       v       v
          ┌────────┐┌──────┐┌──────┐┌──────┐┌──────┐
          │ 全栈   ││ 产品 ││ 后端 ││ 测试 ││ 前端 │
          │Fullstack││Product││Backend││Testing││Frontend│
          └────┬───┘└──┬───┘└──┬───┘└──┬───┘└──┬───┘
               │       │       │       │       │
               v       v       v       v       v
          诊断式对话 → 收集上下文 → 方案设计 → 产出提示词
                                                 │
                                                 v
                                        用户在新会话中执行
                                                 │
                                                 v
                                    ┌────────────────────┐
                                    │     Evaluate       │
                                    │  角色化 8 维评分    │
                                    └────────────────────┘
```

**三条核心原则：**

1. **教练只引导不执行** — 产出诊断结论和提示词，不直接写业务代码
2. **历史资产优先** — 不从空白页开始，先盘点现有代码/文档/数据
3. **标准工作流** — 所有引导遵循：brainstorming → plans → TDD → code review → verification

## 常见问题

**Q: 选了角色之后能换吗？**
说"切换角色"即可。教练会展示选择菜单、更新记忆，并建议开新会话避免上下文干扰。

**Q: 复盘数据从哪来？**
`~/.claude/projects/` 下的会话记录。分析工具使用频率、Skill 调用情况等行为信号，不读你的代码内容。

**Q: 不想走引导流程，能直接问技术问题吗？**
可以。说"快速提问"进入自由问答模式，教练基于你的角色领域知识直接回答。

**Q: 缺少 PRD / 技术方案 / 原型怎么办？**
不阻塞。教练会标记 [待补充]，用已有信息继续推进。缺 PRD 就从代码反推需求，缺原型就用临时方案。

**Q: 教练能处理什么类型的任务？**
任何类型——Bug 修复、新功能、重构、PRD、技术方案、安全修复、数据库迁移、API 对接、性能诊断、紧急故障……教练按角色自适应。

## 设计理念

本项目的核心洞察：

> **大多数人用 Claude Code 的方式是错的。** 他们把它当"代码生成器"——给个需求，等代码出来。结果反复修改、缺少测试、架构混乱。
>
> 正确的方式是把 Claude Code 当"编程搭档"——先诊断、再规划、用 TDD 执行、最后验证。AI Coach 就是教会你这种工作方式的教练。

**为什么是 Skill 而不是 Prompt？**
- Prompt 是一次性的，用完就忘
- Skill 是持久化的，可以记忆角色、引用历史、跨会话衔接
- Skill 可以调用其他 Skill（brainstorming → TDD → code review），形成完整工作流
