## 1. 问题总结

| # | 现状问题 | 影响 |
|---|---------|------|
| 1 | 提问过多（7-8 项收集清单） | 用户疲劳，放弃使用 |
| 2 | 教练给直接方案（表设计、API 规格） | 违背"引导不执行"原则，用户缺少思考 |
| 3 | 无版本检查机制 | 用户可能用过时版本 |
| 4 | 无方法论知识库 | 引导缺乏体系支撑 |
| 5 | 入口复杂（角色选择 → 菜单 → 意图识别） | 启动摩擦大 |

## 2. 设计目标

1. **3-5 轮自适应提问** — 根据任务复杂度动态调整，简单任务 3 轮结束
2. **产出方法论指导报告** — 推荐做事方法，不给具体方案（表/接口/代码）
3. **3 个平行入口** — 问题/脑暴、角色视角、总结评估，降低启动摩擦
4. **方法论知识库** — 可扩展的方法论文件系统，支持场景检索
5. **静默版本检查** — 启动时从 GitHub 检查更新
6. **所有角色统一流程** — 通过 COACH_TEMPLATE 强制一致性

### 4.1 SKILL.md — 主入口路由

**职责**: 3 个平行入口 + 全局规则 + 版本检查触发

#### 入口设计

用户输入 `/ai-coach` 时，使用 AskUserQuestion 展示 3 个入口：

> "你今天想做什么？"

| 入口 | 选项文案 | 路由 |
|------|---------|------|
| 🎯 问题/脑暴 | "我有个任务要做" 或 "我想讨论一个想法" | → 角色匹配 → 角色 SKILL |
| 👤 角色视角 | "用特定角色视角看问题"（后端/前端/产品/测试/全栈） | → 角色选择 → 角色 SKILL |
| 📊 总结评估 | "复盘我的编码过程" | → evaluate SKILL |

**入口 1 "问题/脑暴" 的自动角色匹配：**
- 用户描述任务后，根据关键词自动匹配角色（沿用现有路由表）
- 匹配成功 → "这看起来是后端任务，用后端视角来引导？" + 确认
- 匹配失败 → 默认全栈角色
- Memory 有角色记忆 → 优先复用，"继续用{角色}视角？"

**入口 2 "角色视角" 的角色选择：**
- 如果 Memory 有角色记忆，先确认 "上次用的是{角色}，继续用？"
- 否则展示角色列表（产品/后端/测试/前端 + Other→全栈）

**精简后的全局规则**（从 13 条精简到核心）：

1. **教练不执行** — 只产出方法论推荐和示范提示词
2. **不给具体方案** — 不产出表设计、API 规格、代码架构。只推荐"用什么方法论去设计"
3. **新会话分离** — 引导结束后建议开新会话执行
4. **通用降级** — 缺资料不阻塞，标记[待补充]继续
5. **superpowers 标准流程** — brainstorming → writing-plans → TDD → code review → verification
6. **治具改进循环** — 发现上下文缺口时建议永久修复到 CLAUDE.md/Memory/Hooks

**启动时触发**:
- 调用 init SKILL 做环境自检（含版本检查）
- 检查 Memory 角色记忆

### 4.2 common/COACH_TEMPLATE.md — 标准教练流程模板

**职责**: 定义所有角色必须遵循的统一教练流程

#### 流程结构

```
Phase 1: 理解（1-2 轮）
  ↓
Phase 2: 诊断（1-2 轮）
  ↓
Phase 3: 方法论推荐 + 指导报告输出（1 轮）
```

#### Phase 1: 理解（Round 1-2）

**Round 1 — 任务快照**
一次提问，收集 3 个核心信息：
```
我需要了解 3 件事来帮你：
1. 你要做什么？（一句话描述任务目标）
2. 现在是什么状态？（还没开始 / 做到一半 / 快做完了）
3. 有没有现成资料？（PRD / 设计稿 / 现有代码 / 都没有）
```

**Round 2 — 复杂度判断**（自适应）
根据 Round 1 回答判断复杂度：
- **简单**（CRUD/配置/单点修复）→ 跳过 Phase 2，直接到 Phase 3
- **中等**（多模块/有集成/有现有代码）→ Phase 2 问 1 轮
- **复杂**（跨系统/架构变更/新领域）→ Phase 2 问 2 轮

#### Phase 2: 诊断（Round 2-4，自适应）

每轮 **最多 3 个问题**，由角色 SKILL 定义具体问题池。

模板提供框架：
```
基于你的描述，我还想确认：

1. [角色特定问题 - 由角色 SKILL 提供]
2. [角色特定问题 - 由角色 SKILL 提供]
3. [角色特定问题 - 由角色 SKILL 提供]（可选，仅复杂任务）
```

**退出条件**（满足任一则结束诊断）：
- 已收集足够信息可推荐方法论
- 用户说"差不多了"/"就这些"
- 已达到最大轮数（简单 0 轮，中等 1 轮，复杂 2 轮）

#### Phase 3: 方法论指导报告

**必须输出**，格式固定：

```markdown
---
## 🎯 方法论指导报告

### 任务概要
{一句话总结任务}

### 复杂度评估
{简单/中等/复杂} — {判断依据}

### 推荐方法论
1. **{方法论名称}** — {为什么适用这个任务}
   - 📖 详细参考：knowledge/{文件名}.md
   - 🔑 核心要点：{3-5 个要点}

2. **{方法论名称}**（如果需要组合多个方法论）
   - 📖 详细参考：knowledge/{文件名}.md
   - 🔑 核心要点：{3-5 个要点}

### 执行建议
1. 先用 `/brainstorming` 讨论方案
2. 再用 `/writing-plans` 产出实施计划
3. 编码走 `/test-driven-development`
4. 完成后 `/requesting-code-review`
5. 最终 `/verification-before-completion`

### 示范提示词
> 在新会话中使用：
> "{根据任务定制的示范提示词，引用推荐的方法论要点}"

### ⚠️ 待补充项
- {列出所有标记为[待补充]的信息}
---
```

**硬性约束**（写入模板的红线）：
- ❌ 不产出表结构设计
- ❌ 不产出 API 接口规格
- ❌ 不产出代码架构图
- ❌ 不产出具体代码
- ✅ 只推荐方法论 + 为什么 + 怎么用
- ✅ 只给示范提示词（让用户在新会话中让 AI 去做）

### 4.5 init/SKILL.md — 版本检查

在现有 Layer 1 全局检查中新增第 0 步：

```markdown
### 0. 版本检查（静默）

- 执行 `curl -sf https://raw.githubusercontent.com/tianyiwxgm-netizen/ai-coach/main/SKILL.md | head -5`
- 从返回内容提取 `version:` 字段
- 与本地 SKILL.md 的 version 比较
- **有新版本时**：输出一行 "💡 AI Coach 有新版本 {remote_version}（当前 {local_version}），运行 install.sh 可更新"
- **无更新或网络失败时**：不输出任何内容（静默）
- 超时设置：3 秒，超时则静默跳过
```
