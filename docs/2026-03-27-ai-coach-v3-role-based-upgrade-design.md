# AI Coach v3 角色化升级设计文档

> **日期**：2026-03-27
> **版本**：v2.0.0 → v3.0.0
> **目标**：将 ai-coach 从通用教练升级为角色化教练系统，支持 5 种角色，适配传统公司 AI 编程转型场景

---

## 一、背景与目标

### 业务背景

传统公司正在使用 Claude Code 进行 AI 编程转型。领导要求：不能立刻让所有人变成全栈，先让大家按照原有角色（产品、后端、测试、前端）使用 AI 提效。

### 升级目标

1. ai-coach 从"通用教练"升级为"角色化教练"，支持 5 种角色
2. 每个角色有独立的工作流、skill 推荐、评估维度
3. 主 SKILL.md 保持精简，通过子 skill 按需加载角色内容
4. 首次使用时自检环境并自动配置
5. 教练只引导不执行，产出引导方案和示范提示词

### 设计原则

- **使用上简单，内部可以复杂**：用户只需 `/ai-coach`，内部路由和子 skill 调用对用户透明
- **教练不下场**：只产出"做什么、怎么做、用哪个 skill"，不执行业务任务（写代码、修 Bug 等）。唯一例外：init 阶段可执行环境配置和工具安装
- **上下文精准**：选了哪个角色就只加载那个角色的内容，避免信息过载
- **历史资产优先**：所有角色的工作都要建立在历史 PRD、代码、数据库基础上

├── product/
│   └── SKILL.md                 # 子 skill：产品角色
│
├── backend/
│   └── SKILL.md                 # 子 skill：后端角色
│

│   └── SKILL.md                 # 子 skill：测试角色

### 为什么选方案 A+

| 决策 | 选择 | 理由 |
|------|------|------|
| 架构 | 单入口 + 子 skill | 用户只记 `/ai-coach`，体验简单 |
| 文件加载 | Skill 工具加载子 skill | 指令权重正确（vs Read 加载普通文件权重不足） |
| 角色记忆 | Memory 存储 + 菜单重选 | 日常零成本，切换也方便 |
| 自检 | 三层静默 + 自动修复 | 不吓人，不阻塞 |
| 评估 | 角色定制维度 | 公平，有针对性 |
| 教练边界 | 只引导不执行 | 上下文不爆，职责清晰 |

## 三、入口流程（SKILL.md）

### 核心逻辑

```
用户输入 "/ai-coach" 或 "教练"
        │
        ▼
┌─ 检查 Memory 中是否有已选角色 ─┐
│                                │
│  有角色记忆                     无角色记忆（首次）
│       │                              │
│       ▼                              ▼
│  检查是否带任务描述              调用 ai-coach-init 子 skill
│       │                         （静默自检 + 自动修复）
│   有描述    无描述                     │
│    │          │                       ▼
│    │          ▼                  展示角色选择菜单
│    │     展示菜单：                    │
│    │     1. 开启新任务                 ▼
│    │     2. 复盘总结              用户选角色 → 存入 Memory
│    │     3. 切换角色                   │
│    │     4. 快速提问                   ▼
│    │          │               角色专属自检（静默）
│    ▼          ▼                        │
│  直接路由到对应角色子 skill      ◄──────┘
│  （快捷路由，跳过菜单）
└────────────────────────────────────────┘
```

### 快捷路由原则

- 带任务描述 → **0 步**到干活（自动识别角色流程入口）
- 无描述老用户 → **1 步**到干活（菜单选择）
- 首次用户 → 选角色 + 静默自检（对用户感知只有 **1 步**）

### 全局规则（所有角色继承）

1. **教练不执行业务任务**：只产出引导方案和示范提示词，不执行写代码、修 Bug 等业务工作。例外：init 阶段可执行环境配置和工具安装
2. **新会话分离**：引导结束后建议用户开新会话执行
3. **语言自适应**：默认中文，全英文输入时切英文
4. **中途切换**：任何步骤可切换角色或任务类型
5. **角色记忆**：选择的角色存入 Memory，下次自动加载
6. **通用降级原则**：当用户无法提供某项资料时，教练不阻塞流程，而是标记为[待补充]并提供替代收集方法（如从代码反推需求、从接口反推文档），用已有信息继续推进
7. **角色切换策略**：切换角色时更新 Memory 中的角色记忆（覆盖式），建议用户开新会话（避免旧角色上下文残留）。同一会话内切换会导致上下文累加
8. **快速提问模式**：用户选"快速提问"后进入自由问答，教练基于当前角色领域知识回答，不走引导流程。结束后提示回到主菜单
9. **子 skill description 精简**：所有子 skill 的 SKILL.md description 字段控制在一行以内，避免 Claude Code 启动时上下文膨胀

## 四、首次自检初始化（init/SKILL.md）

### 触发时机

| 场景 | 触发内容 |
|------|---------|
| 首次使用 ai-coach | 全局检查 + 角色专属检查 |
| 首次切换到新角色 | 仅该角色的专属检查 |
| 日常使用 | 静默快检（有问题才提醒） |

### Layer 1：全局检查

| # | 检查项 | 缺失时处理 | 用户看到的提示 |
|---|--------|-----------|---------------|
| 1 | CLAUDE.md 存在 | 引导 `/init` | "我需要了解一下你的项目，帮你生成一个项目说明文件" |
| 2 | CLAUDE.md 含关键段落 | 提示补充 | "项目说明文件缺少一些关键信息，我来帮你补充" |
| 3 | Memory 目录 | 自动创建 | （静默） |
| 4 | superpowers skill | 提示安装 | "需要安装一个核心工具包，我来帮你装？" |
| 5 | 推荐配置 | 列出建议（不强制） | "以下配置可以提升效率，建议有空时设置" |

推荐配置（第 5 项展开）：
- LSP 插件（代码实时检查）
- `CLAUDE_CODE_SUBAGENT_MODEL=sonnet`（子代理用更快模型省成本）
- 常用命令权限白名单（避免每次运行测试点确认）

### Layer 2：角色专属检查

**产品**：prd-development skill、user-story skill、jobs-to-be-done skill（推荐）、docs/ 目录

**后端**：构建命令可执行、测试框架可运行、数据库连接配置

**测试**：Playwright 已安装、Playwright MCP 已配置、webapp-testing skill、tests/ 目录

**前端**：前端构建工具可运行、Playwright（自测用）、测试框架可运行

**一人全栈**：后端 + 前端检查项合集

### Layer 3：日常快检

完全静默执行。只检查 CLAUDE.md 在不在、角色记忆在不在。有问题才输出一行提醒。

### 输出格式

全部通过：`✓ 环境检查完成，一切就绪。`

有缺失：
```
环境检查完成。有 2 项需要配置：
1. 需要安装 PRD 写作工具 → 我来帮你装？(Y/n)
2. 建议建立文档目录 → 我来创建？(Y/n)
其余 6 项检查均正常。
```

## 五、一人全栈角色（fullstack/SKILL.md）

### 定位

现有 ai-coach SKILL.md 的核心内容升级版。作为默认角色，覆盖全类型任务。

### 相对现有版本的升级点

| 现有 | 升级 |
|------|------|
| 推荐工作流只列 brainstorm/write-plan/execute-plan | 精准映射到 superpowers 具体 skill |
| 示范提示词模板缺"验证方式" | 模板增加【验收验证】段 |
| 无上下文管理提醒 | 引导结束时提醒上下文窗口管理 |
| 零资料降级只有文字指导 | 增加"5 分钟快速收集法" |
| enterprise 子 skill 混合产品+技术 | 角色专属内容移到各角色子 skill |

### 核心流程

```
Step 1: 意图识别（保留快捷路由 + 主菜单）
Step 2: 任务诊断（保留复杂度判断 + 领域专项触发）
Step 3: 通用引导
  ├── 3.1 上下文收集（四维度 + 升级零资料降级）
  ├── 3.2 方法论推荐（升级 skill 精准映射表）
  ├── 3.3 示范提示词生成（升级模板，加验收验证段）
  └── 3.4 上下文管理提醒（新增）
```

### Skill 精准映射表

| 任务类型 | 推荐 Skill 组合 |
|---------|-----------------|
| 新功能开发 | brainstorming → writing-plans → executing-plans → verification-before-completion |
| Bug 修复 | systematic-debugging → test-driven-development → verification |
| 紧急故障 | 直接定位修复 → 事后补 systematic-debugging + TDD |
| 重构 | brainstorming → writing-plans → TDD 驱动执行 |
| PRD/方案/文档 | brainstorming → 新会话用 /prd-development |
| 性能优化 | systematic-debugging（定位瓶颈）→ TDD（验证优化） |
| 安全修复 | systematic-debugging → TDD + 全局扫描 |

### 升级后的示范提示词模板

```
"我需要{任务描述}。

【背景】{业务背景或项目背景}
【现状】{当前状态描述}
【目标】{期望达到的效果和验收标准}
【约束】{技术限制、时间要求等}
【资料】{相关文档/代码路径/截图/日志}
【验收验证】{具体的验证方式：跑什么测试、检查什么页面、对比什么数据}

请{具体的行动指令}。"
```

### 保留的核心内容

- 教练内功心法（五大核心原则、六个心智跃迁、复杂度阶梯）
- 全局规则（中途切换、无关问题、回退、语言自适应）
- Step 0 经验检测
- Bug 修复铁律、安全修复三原则、重构铁律
- 零资料降级策略
- 复杂度放大原则（企业级/巨型级追加）

## 六、产品角色（product/SKILL.md）

### 定位

引导产品经理使用 Claude 产出**可直接交付给开发的 PRD + 可演示原型**。核心差异：建立在历史系统、历史 PRD、现有代码基础上。

### 流程

```
Step P1: 意图识别
  - 写新 PRD / 补充现有 PRD / 需求分析 / 复盘

Step P2: 历史资产盘点（核心差异环节）
  收集项：
  1. 现有系统功能（截图/录屏/线上地址）
  2. 历史 PRD/需求文档（文档路径）
  3. 现有数据库表结构（Entity/Schema）
  4. 现有接口（Controller/Swagger）
  5. 线上运营数据（日活/订单量/转化率）
  6. 用户反馈（客服工单/调研报告）

Step P3: 业务分析与需求洞察（6 个子环节，按需组合）
  P3.1 商业分析 — 为什么做？商业目标？量化标准？
  P3.2 用户分析 — 谁在用？什么场景？痛点？Jobs-to-be-Done
  P3.3 竞品分析 — 竞品/替代方案？差异化？
  P3.4 产品能力差距分析 — 现有 vs 目标，差距矩阵
  P3.5 需求列表与优先级 — MoSCoW 优先级排序
  P3.6 业务沟通准备 — 与业务方对齐的沟通提纲

Step P4: PRD 产出引导
  - PRD 结构：10 章（执行摘要→问题陈述→用户→战略→方案→指标→详细需求→非功能→不做的事→依赖风险）
  - AI 可执行 PRD 标准：
    | 维度 | 概要 PRD（不够） | AI 可执行 PRD（目标） |
    |------|-----------------|---------------------|
    | 需求描述 | "支持批量导入" | 用户故事 + 验收标准 + 边界情况 |
    | 交互设计 | "用户上传文件" | 逐步交互流程 |
    | 数据字段 | "商品信息" | 完整字段列表（名/类型/必填/校验/默认值） |
    | 接口契约 | 无 | 输入/输出/错误码/状态流转 |
    | 异常处理 | 无 | 每个环节的异常场景 + 处理方式 |
  - 产出示范提示词，引导产品在新会话中用 /prd-development 执行

Step P5: 原型产出引导
  路径 A：HTML/CSS 原型（Claude 生成，Tailwind 快速美化，页面间可跳转）
  路径 B：线框图 + 布局文档（Claude 生成 Mermaid 流程图 / ASCII 线框图 / HTML 模拟线框，配合页面布局描述文档。适合有设计师配合的团队，Claude 产出布局描述，设计师负责视觉稿）

Step P6: 交付物检查与开发交接
  交付物清单：
  [ ] PRD 文档（AI 可执行标准）
  [ ] 原型（可在浏览器打开或查看的设计图）
  [ ] 能力差距矩阵
  [ ] 需求优先级清单

  交接指引：
  - 后端：用 /ai-coach 后端角色，将 PRD 路径作为输入
  - 测试：用 /ai-coach 测试角色，将 PRD + 技术方案 + 开发代码作为输入
  - 前端：用 /ai-coach 前端角色，将 PRD + 原型 + 后端接口作为输入
```

### 推荐 Skill

| 环节 | 推荐 Skill（新会话中使用） |
|------|--------------------------|
| 业务分析 | /brainstorming + /deep-think |
| 用户研究 | /jobs-to-be-done + /user-story-mapping |
| PRD 编写 | /prd-development + /user-story |
| 竞品分析 | /brainstorming（引导 Claude 搜索整理） |

## 七、后端角色（backend/SKILL.md）

### 定位

引导后端开发者完成**技术方案设计 → 编码实现 → 自测交付**全流程。偏全栈：后台管理系统页面 100% 由后端负责。

### 流程

```
Step B1: 意图识别
  - 新功能开发 / Bug 修复 / 紧急线上故障 / 大型项目规划 / 复盘

Step B2: 技术上下文收集
  收集项：
  1. 产品 PRD（必须）
  2. 原型/设计稿
  3. 现有代码结构（涉及模块）
  4. 现有数据库表
  5. 现有 API 接口
  6. 历史技术方案
  7. 线上数据库现状（数据量、需注意的数据）

Step B3: 技术方案设计引导
  方案结构：需求摘要 → 现有架构分析 → 方案设计（表结构+接口+后台页面+核心逻辑）→ 技术选型 → 影响范围 → 测试策略 → 风险与降级

  后台管理页面特别引导：
  - 每个页面：页面名 + 功能描述 + 字段列表 + 操作按钮 + 查询条件
  - 遵循项目现有页面风格

Step B4: 实施计划引导
  任务拆分：
  Phase 1: 数据层（表结构 + Mapper + 单元测试）
  Phase 2: 服务层（Service + 业务逻辑 + 单元测试）
  Phase 3: 接口层（Controller + 集成测试）
  Phase 4: 页面层（Thymeleaf 模板 + 页面测试）
  Phase 5: e2e 测试（全链路验证）
  Phase 6: 自测 + 修复 + 交付

Step B5: 编码执行引导
  必须遵循的规则：
  1. TDD 铁律（superpowers:test-driven-development）
  2. 小步提交
  3. 后台页面也要有测试（MockMvc）
  4. 遇 Bug 用 superpowers:systematic-debugging
  5. 交付前用 superpowers:verification-before-completion

Step B6: 自测与交付引导
  自测检查清单：
  [ ] 单元测试（核心逻辑 100% 覆盖）
  [ ] 集成测试（所有接口正常+异常+权限）
  [ ] 后台页面测试（渲染+表单+列表）
  [ ] e2e 全链路验证（页面→数据库的完整流程）
  [ ] 构建验证（./gradlew build 通过）
  [ ] 数据验证（表结构+测试数据+迁移脚本）

Step B7: 交付物检查与交接
```

### 特殊流程

**B-Bug（Bug 修复）**：收集信息 → 引导 systematic-debugging 根因分析 → 引导 TDD 修复

**B-Emergency（紧急故障）**：快速定位 → 止血方案（回滚 or 定点修复）→ 事后补测试+复盘

**B-Large（大型项目）**：调用 enterprise 知识库 → 全景架构分析 → 分阶段规划 → 每阶段走标准流程

### 推荐 Skill

| 环节 | 推荐 Skill（新会话中使用） |
|------|--------------------------|
| 技术方案 | /brainstorming → /writing-plans |
| 编码执行 | /executing-plans + /test-driven-development |
| Bug 修复 | /systematic-debugging → /test-driven-development |
| 交付验证 | /verification-before-completion |

## 八、测试角色（testing/SKILL.md）

### 定位

引导测试工程师实现 AI 驱动的**全流程自动化测试**：测试用例设计 → 自动化执行 → 测试报告生成。

### 输入依赖

| 输入 | 来源 | 必须/可选 |
|------|------|----------|
| 产品 PRD | 产品角色产出 | 强烈推荐（缺失时从代码反推需求） |
| 技术方案 | 后端角色产出 | 推荐（缺失时从代码提取架构） |
| 开发代码 | 后端/前端代码仓库 | 必须 |
| 原型/设计稿 | 产品角色产出 | 可选（e2e 测试对照用） |
| API 接口文档 | 技术方案 | 推荐（缺失时从 Controller 提取接口列表） |

### 流程

```
Step T1: 意图识别
  - 新功能全量测试 / 回归测试 / 单接口测试 / 测试体系建设 / 复盘

Step T2: 测试上下文收集
  收集项：PRD、技术方案、开发代码、API 接口清单、测试环境、现有测试代码、历史 Bug 记录

Step T3: 测试策略规划
  产出：测试范围 + 测试金字塔分配 + 测试数据策略 + 环境要求

Step T4: 测试用例设计引导（三层）
  T4.1 单元测试用例：
    | 用例ID | 测试对象 | 场景 | 输入 | 期望输出 | 优先级 |
    每个 PRD 验收标准至少对应 1 个测试用例

  T4.2 接口测试用例：
    每个接口：正常场景 + 异常场景 + 边界场景
    含请求体、期望状态码、期望响应

  T4.3 e2e 测试用例：
    每个用户流程：前置条件 + 逐步操作 + 每步验证点
    关联 PRD 需求编号

Step T5: 自动化测试执行引导
  T5.1 单元+集成测试（TDD 方式）
    JUnit 5 + Mockito + MockMvc，superpowers:test-driven-development

  T5.2 e2e 测试（Playwright）
    TypeScript，语义定位器优先，Playwright MCP 对照真实页面验证选择器
    每步截图存证，失败自动截图+控制台日志

  T5.3 Playwright 三代理模式（高级，测试体系成熟后引导）
    Planner Agent → Generator Agent → Healer Agent

Step T6: 测试报告生成引导
  标准化报告结构：
  - 基本信息 + 测试概览表 + 失败用例详情 + 缺陷清单 + PRD 需求覆盖率 + 测试结论

Step T7: 缺陷管理与回归
  发现缺陷 → 记录 → 提交给开发 → 开发修复后回归 → 更新报告 → 发布确认
```

### 特殊流程

**T-Quick（单接口快速测试）**：跳过完整收集，快速生成正常+异常+边界用例，产出示范提示词

**T-Regression（回归测试）**：git diff 查变更 → 影响分析 → 执行回归 → 回归报告

## 九、前端角色（frontend/SKILL.md）

### 定位

引导前端开发者完成**客户端应用的技术方案设计 → 编码实现 → 自测交付**。覆盖 Web 前端、移动端（安卓/苹果）、小程序等平台。不绑定技术栈，在 Step F2 通过"目标平台"判断路由到对应的技术选型和适配策略。

### 输入依赖

| 输入 | 来源 | 必须/可选 |
|------|------|----------|
| 产品 PRD | 产品角色产出 | 必须 |
| 原型/设计稿 | 产品角色产出 | 必须 |
| 技术方案 | 后端角色产出 | 必须 |
| 后端 API 接口文档 | 后端开发完成 | 必须 |
| 历史前端代码 | 代码仓库 | 有则必须 |

### 流程

```
Step F1: 意图识别
  - 新功能开发 / Bug 修复 / 紧急故障 / 大型项目 / 复盘

Step F2: 前端上下文收集
  收集项：PRD、原型、后端接口文档、目标平台、技术栈、现有前端代码、现有页面截图、设备兼容要求

Step F3: 前端技术方案设计引导
  方案结构：
  - 需求摘要 → 技术栈确认 → 现有架构分析 → 页面设计表 → 组件拆分表
  - 接口对接方案 → 状态管理设计 → 设备适配策略 → 测试策略

  技术选型引导（未定时）：
  | 场景 | 推荐 |
  |------|------|
  | Web SPA + 团队熟 JS | Vue / React |
  | Web + 快速原型 | Next.js / Nuxt.js |
  | 双端移动+团队熟 JS | React Native |
  | 双端移动+高性能 | Flutter |
  | 单端+深度原生 | Swift / Kotlin |
  | 轻量移动+快速迭代 | H5 (WebView) |
  | 小程序 | 微信原生 / Taro / uni-app |

Step F4: 实施计划引导
  Phase 1: 项目基础（脚手架/网络层/状态管理/路由）
  Phase 2: 基础组件（TDD 逐个开发）
  Phase 3: 页面开发（静态→交互→接口，每页 TDD）
  Phase 4: 接口联调（mock→真实）
  Phase 5: 设备适配（多尺寸/低端设备/系统版本）
  Phase 6: e2e 测试 + 交付

Step F5: 编码执行引导
  规则：TDD 铁律 + 原型对照 + 接口 Mock 优先 + 小步提交
  遇 Bug 用 systematic-debugging（先区分前端 vs 后端问题）

Step F6: 自测与交付引导
  检查清单：
  [ ] 组件测试（渲染+交互）
  [ ] 页面测试（渲染+导航+表单）
  [ ] 接口联调（请求/响应/错误处理/加载状态）
  [ ] 视觉还原（逐页面对照原型截图）
  [ ] 设备兼容（目标设备清单全验证）
  [ ] e2e 全链路
  [ ] 构建验证（安装包可正常运行）
  [ ] 性能基线（首屏/滚动/内存）

Step F7: 交付物检查与交接
```

### 特殊流程

**F-Bug**：收集信息（含设备/系统版本）→ 问题分类（UI/交互/接口/兼容/性能）→ 引导 systematic-debugging + TDD

**F-Emergency**：止血方案（回滚版本/热更新/服务端降级/定点修复）→ 事后补测试+复盘

**F-Large**：前端架构设计（模块拆分/路由/组件库/状态管理）→ 分阶段交付

| 环节 | 推荐 Skill（新会话中使用） |
|------|--------------------------|
| 技术方案 | /brainstorming → /writing-plans |
| 编码执行 | /executing-plans + /test-driven-development |
| e2e 测试 | /webapp-testing（Playwright/Detox/Appium） |
| Bug 修复 | /systematic-debugging → /test-driven-development |

## 十一、enterprise 知识库精简

### 变化

现有 enterprise/SKILL.md 包含产品知识库 + 技术知识库 + 巨型项目知识库。

升级后：
- **产品知识库** → 移入 product/SKILL.md
- **技术知识库（新功能/Bug/重构）** → 移入 backend/SKILL.md
- **enterprise 保留**：巨型项目全链路框架、跨团队协调、风险评估、分阶段路线图模板、Strangler Fig 策略、评估驱动开发

enterprise 变为纯粹的"跨角色通用大型项目知识库"，任何角色遇到大型项目时调用。
