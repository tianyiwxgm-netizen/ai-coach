# Case 15: 评估得分低 - 用户不满评分

## 基本信息

- **分类**: 评估流程 / 情绪处理 / 评分争议
- **优先级**: P1
- **可能翻车的点**: evaluate 产出评分后（假设 L1 / 25 分），用户觉得不公平并表达不满。SKILL.md 是否有处理"用户质疑评分"的分支？教练可能出现两种极端：(1) 坚持评分不解释，让用户更生气；(2) 为了安抚用户随意调高分数，失去评估公信力。正确做法应该是解释评分依据并引导改进。

## 真实用户输入

第一步：
```
/ai-coach "复盘一下今天的工作"
```

第二步（看到评分后）：
```
这个评分太离谱了，我明明写了很多代码
```

## SKILL.md 应该怎么处理

1. 正常完成评估流程，产出评分报告
2. 收到用户质疑后，不修改评分，但详细解释每个维度的评分依据
3. 指出"写了很多代码"和"高质量工程实践"之间的区别（如：代码量 vs 测试覆盖率、TDD 使用等）
4. 将质疑转化为改进方向，给出具体可操作的提升建议

## 验收标准

- [x] AC1: 评估报告本身包含每个维度的评分依据（不是只给总分）
- [ ] AC2: 面对用户质疑，不直接修改评分以安抚情绪
- [ ] AC3: 解释评分时引用具体的会话证据（如"在 X 会话中没有看到测试先行的做法"）
- [ ] AC4: 语气共情但不卑微（如"理解你的感受，让我解释一下评分依据"）
- [x] AC5: 将低分维度转化为可操作的改进建议（不是笼统的"要加油"）
- [ ] AC6: 不出现防御性语气（如"评分是客观的，你不应该质疑"）
- [ ] AC7: 如果用户提供了评估遗漏的证据，愿意补充考量并重新评估
