# Case 09: 用户想复盘，但从未用过任何 Skill

## 基本信息

- **角色**: 后端开发（或任何角色）
- **场景**: 用户听说可以复盘，想看看自己的 AI 编程水平。但他平时只用 Claude Code 聊天写代码，从没用过 /brainstorming、/test-driven-development 等任何 Skill
- **前置条件**: Memory 中已有 `AI教练角色: 后端`，有一定量的会话记录

## 真实用户输入

```
复盘
```

## SKILL.md 应该如何处理

1. "复盘"匹配主 SKILL.md 的路由 → 调用 ai-coach-evaluate
2. evaluate Step 9 角色路由 → 后端 → Step 13-C
3. Step 10 选择分析范围 → 用户选"最近一周"或"全部历史"
4. Step 10.2 运行 analyze_sessions.py → 产出汇总信号
5. Step 11-12 提取行为信号和上下文工程检查
6. Step 13-C 后端 8 维评分

## 验收标准

- [ ] 评估报告不应该让用户觉得被"审判"或"打击"
- [ ] 对于从未用过 Skill 的用户，低分维度的措辞应该是鼓励性的（"这是你下一步可以探索的方向"）
- [ ] 应该区分"不知道有这些工具"和"知道但不用"两种情况
- [ ] 改进建议应该具体可操作，不要只说"建议多用 TDD"
- [ ] 如果用户所有 Skill 使用都是 0，至少有一些维度能基于其他信号给出合理评分（不全是 0 分）

## 可能翻车的点

1. **多个维度集中低分**：后端 8 维评分中，C1 TDD 合规度需要 test_command_count 和 TDD Skill 使用信号——如果用户从未运行过测试命令，0-20 分段。C4 SOP 合规需要规划 Skill——从未用过，0-20 分段。C7 Bug 修复规范需要 systematic-debugging 使用——从未用过，0-20 分段。三个维度集中低分，总分可能只有 L1-L2，用户看到会很沮丧
2. **"SOP 合规度"和"Skill 使用熟练度"对新手不友好**：这两个维度本质上是在评估"你有没有用我的工具"，对从未接触过 Skill 体系的用户来说不公平。评估标准暗含了"使用 Skill 是正确做法"的预设
3. **A8/C8 方法论内化度的评估依据不足**：用户可能是一个很好的开发者，只是不用 Skill。但"方法论内化度"的评分信号（ailearn 五大原则体现、复杂度阶梯思维）需要从用户消息中定性分析。如果用户消息都是简短指令（"帮我写个接口"），AI 很难判断他的方法论水平
4. **评估报告可能显得自说自话**：报告中的改进建议都是"用 /brainstorming"、"用 /test-driven-development"——全是推荐自家产品。用户可能觉得"你就是想让我多用你的 Skill"
5. **置信度标记可能被忽视**：Step 10.4 有数据充分度判断（1-2 个会话标记置信度"低"），但如果用户有 20+ 个会话但 Skill 使用全是 0，置信度会标"高"。高置信度 + 低分数 = 更打击人
6. **缺少基线参考**：用户看到 45/100 (L2) 不知道这个分数意味着什么。没有"同等经验的开发者平均分"或"你只需要做 X 就能到 L3"这样的上下文信息
7. **历史趋势为空**：首次评估没有历史对比，学习成长维度无法给出有意义的评分

### 验收标准逐项检查

- [ ] ❌ 评估报告不应该让用户觉得被"审判"或"打击"
  - **依据：** ai-coach-evaluate SKILL.md Step 14 输出格式中使用状态标记："分数 < 40：!! 最需改进"。当多个维度集中低分时（如 C1 TDD 合规 0-20 + C4 SOP 合规 0-20 + C7 Bug 修复规范 0-20），输出会出现多个"!! 最需改进"标记。虽然 Step 14 的 TOP 3 改进建议和 Step 16 教学引导有正向引导，但**评分输出本身没有任何缓冲或鼓励性措辞的条文**。星标映射中 0-20 显示为"★☆☆☆☆"，多个一星评分视觉上很打击人。SKILL.md 没有"当多个维度集中低分时使用鼓励性措辞"的条文。
- [ ] ❌ 对于从未用过 Skill 的用户，低分维度的措辞应该是鼓励性的（"这是你下一步可以探索的方向"）
  - **依据：** Step 14 的状态标记是机械化的规则（分数 < 40 = "!! 最需改进"），没有根据低分的原因（"不知道有这些工具"vs"知道但不用"）调整措辞。Step 16 教学引导有正向引导（"为什么重要"+"正确做法示范"），但 Step 14 终端摘要是用户首先看到的，第一印象已经是大量"!! 最需改进"。SKILL.md 没有"首次评估使用特殊措辞"或"Skill 使用为 0 时用探索性而非批评性措辞"的条文。
- [ ] ❌ 应该区分"不知道有这些工具"和"知道但不用"两种情况
  - **依据：** 评分规则中，C1 TDD 合规度的 0-20 分段是"从未运行测试"，C4 SOP 合规的 0-20 分段是"所有会话直接写码"，C7 Bug 修复规范的 0-20 分段是"遇 Bug 直接改码，无分析"。这些评分规则**不区分原因**——"从未运行测试"可能是不知道有 TDD 工具，也可能是知道但刻意不用。评分公式是定量的，基于行为计数，不考虑意图。SKILL.md 没有"根据用户历史判断是否知道这些工具"的条文。
- [x] ✅ 改进建议应该具体可操作，不要只说"建议多用 TDD"
  - **依据：** Step 16 教学引导明确要求三层引导："为什么重要"+"你当前的问题（基于会话数据指出具体问题）"+"正确做法示范（具体操作步骤和示范提示词）"+"推荐工具"。后端角色引导要点表中也有具体建议，如"Red-Green-Refactor 铁律，每个 Phase 先写测试"和推荐的 Skill 名称。这些条文保证了建议的具体性和可操作性。
- [x] ✅ 如果用户所有 Skill 使用都是 0，至少有一些维度能基于其他信号给出合理评分（不全是 0 分）
  - **依据：** 后端 8 维评分中，C2 技术方案质量可以基于"编码前方案/Plan Mode"给分，C3 自测交付标准可以基于"verification_skill_usage + 单元测试运行"给分，C5 代码提交质量可以基于"git_commit_count"给分，C6 上下文工程可以基于"CLAUDE.md 存在+Memory+Skills"给分。这些维度不完全依赖 Skill 使用，即使 Skill 使用全为 0，其他信号（如 git 提交频率、CLAUDE.md 完整度、测试命令执行次数）仍然可以贡献分数。所以不会全是 0 分。

### 发现的 SKILL.md 缺陷

1. **评分输出缺少情感缓冲**：Step 14 的状态标记和星标是机械化的，当多个维度集中低分时视觉冲击强烈。需要添加"首次评估/大量低分时使用鼓励性包装"的条文。
2. **评分规则不区分原因**：0-20 分段的描述是行为层面的（"从未运行测试"），不考虑原因（"不知道可以这样做"vs"刻意不做"）。建议在低分段增加"如果用户首次评估且 Skill 使用为 0，措辞改为探索性引导"。
3. **C4 SOP 合规和 C7 Bug 修复规范对新手不公平**：这两个维度本质上评估的是"你有没有用我推荐的工作流"，对完全不了解 Skill 体系的用户来说是零起点评估，公平性存疑。
4. **首次评估缺少基线说明**：用户看到 45/100 (L2) 不知道这意味着什么。Step 14 没有"首次评估的分数解读"或"大多数初次用户处于 L1-L2"的上下文信息。
