# Case 04: 产品经理第一次写 PRD，不懂什么是"历史资产盘点"

## 基本信息

- **角色**: 产品经理
- **场景**: 产品经理第一次用 AI Coach，想写一个新功能的 PRD。她知道要做什么，但完全不了解 AI Coach 的流程和术语
- **前置条件**: Memory 中已有 `AI教练角色: 产品`

## 真实用户输入

```
我要写个新功能的 PRD，就是给门店加个会员系统
```

## SKILL.md 应该如何处理

1. 快捷路由匹配 "写 PRD" → 路由到 P2 历史资产盘点
2. P2 收集清单 6 项：现有系统功能、历史 PRD、数据库表结构、现有接口、运营数据、用户反馈
3. 逐项询问，每项给 3 秒决策窗口，说"没有"就标记 [待补充] 继续
4. 收集完成后进入 P3 业务分析（P3.1 商业分析 → P3.2 用户分析 → ...）
5. 最终 P4 产出 PRD 引导方案和示范提示词

## 验收标准

- [ ] 产品经理能理解每个收集项在问什么（用非技术语言）
- [ ] "数据库表结构"和"现有接口"这类技术术语有通俗解释或替代表述
- [ ] 如果 6 项全说"没有"或"不知道"，流程不卡住，能平滑降级
- [ ] P3 业务分析的问题不要太学术化（JTBD、MoSCoW 等术语需要解释）
- [ ] 最终产出的示范提示词是产品经理能直接复制使用的，不需要填很多技术细节

## 可能翻车的点

1. **P2 术语过多**：6 项收集清单中，第 3 项"数据库表结构"和第 4 项"现有接口"对产品经理来说是完全的技术术语。引导问题"涉及哪些数据库表？有 Entity 类或 Schema 文件吗？"——产品经理不知道 Entity 和 Schema 是什么
2. **第 5 项"运营数据"太宽泛**：引导问题"日活、订单量、转化率等关键数据能提供吗？"——产品经理可能知道要看数据，但不知道去哪找（BI 系统？后台？问运营？），"数据报表截图 / BI 系统链接 / 运营周报"这些获取方式建议对新手来说不够具体
3. **P3 流程太长**：P3 有 6 个子环节（商业分析、用户分析、竞品分析、能力差距、需求列表、业务沟通准备），全走一遍对第一次使用的产品经理来说信息量太大
4. **JTBD/MoSCoW 等框架术语**：P3.2 直接用 JTBD（Jobs-to-be-Done），P3.5 用 MoSCoW 方法。产品经理如果不了解这些框架，看到"功能性需求 / 情感性需求 / 社会性需求"会困惑
5. **零资料降级策略的"全新项目"分支**：P2 零资料降级有"全新项目 → 跳过历史资产盘点"。但用户说的是"给门店加个会员系统"，这不是全新项目，是在现有系统上加功能，但她可能不知道现有系统的技术细节。这个降级路径不完全适用
6. **治具优先检查可能输出不相关信息**：P2 开头说"盘点前先检查 Memory 有无上次 PRD 决策"——首次使用的产品经理 Memory 里什么都没有，这个检查结果如何呈现？

### 验收标准逐项检查

- [ ] ❌ 产品经理能理解每个收集项在问什么（用非技术语言）
  - **依据：** ai-coach-product SKILL.md P2 收集清单第 3 项引导问题："涉及哪些数据库表？有 Entity 类或 Schema 文件吗？"——"Entity 类"和"Schema 文件"是纯技术术语，产品经理不会知道这是什么。第 4 项："当前有哪些相关 API？有 Swagger 文档吗？"——"API"和"Swagger"也不是产品经理日常使用的词汇。SKILL.md 没有针对产品角色的用户提供通俗化表述。ai-coach-init 的"语言风格"部分有非技术语言要求，但那只适用于 init 阶段，产品角色子 skill 的 P2 收集清单没有做通俗化处理。
- [ ] ❌ "数据库表结构"和"现有接口"这类技术术语有通俗解释或替代表述
  - **依据：** 直接看 P2 收集清单，第 3 项"数据库表结构"和第 4 项"现有接口"都没有提供通俗化的替代表述。获取方式建议也是技术化的："Entity 文件路径 / 数据库 Schema 导出 / CLAUDE.md 中表前缀"和"Controller 文件路径 / Swagger URL / 接口列表文档"。SKILL.md 中没有任何条文要求在产品角色的上下文收集中对技术术语做解释。
- [x] ✅ 如果 6 项全说"没有"或"不知道"，流程不卡住，能平滑降级
  - **依据：** P2 引导方式明确写道："逐项询问，每项给 3 秒决策窗口。用户说'没有'就标记 [待补充] 并继续。不阻塞流程。"零资料降级策略也覆盖了三种情况（全新项目/只有线上系统/有代码没文档）。全局规则第 6 条也提供了通用降级保障。这一条有明确条文支持。
- [ ] ❌ P3 业务分析的问题不要太学术化（JTBD、MoSCoW 等术语需要解释）
  - **依据：** P3.2 直接使用"JTBD（Jobs-to-be-Done）核心思维"标题，虽然括号里有英文全称，但没有用通俗语言解释什么是 JTBD。问题本身用了"功能性需求/情感性需求/社会性需求"这样的学术分类。P3.5 使用"MoSCoW 方法"，虽然表格中解释了每个级别的含义（Must=必须有），但"MoSCoW"这个术语本身没有被解释（它是首字母缩写）。SKILL.md 没有要求在使用框架术语时做通俗化解释。
- [x] ✅ 最终产出的示范提示词是产品经理能直接复制使用的，不需要填很多技术细节
  - **依据：** P4 示范提示词模板中的占位符（{P2 收集的功能清单/截图路径}、{P3.1 商业分析结论}等）都是由教练根据前序步骤收集的信息来填充的（"教练根据前序步骤收集的信息，组装如下提示词"），不是让用户自己填技术细节。产品经理收集的是功能截图、业务背景等她能理解的内容，教练负责组装。这个设计是合理的。

### 发现的 SKILL.md 缺陷

1. **P2 收集清单对产品角色用户不友好**：第 3、4 项使用纯技术术语（Entity/Schema/Controller/Swagger/API），没有提供"产品经理版"的通俗表述。建议：第 3 项可改为"这个功能涉及存储哪些数据？如果不清楚技术细节，可以描述业务上需要管理什么信息"。
2. **P3 框架术语未做通俗化**：JTBD、MoSCoW 等专业框架直接使用缩写，对不了解这些方法论的产品经理不友好。建议在首次出现时用一句话通俗解释。
3. **零资料降级的边界情况**："给门店加会员系统"既不是"全新项目"也不是"有代码没文档"，而是"在现有系统上加功能但不了解技术细节"。这种常见场景没有精确匹配的降级路径。
