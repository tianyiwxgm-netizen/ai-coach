# Case 02: 后端开发选了"产品"角色，然后要写接口

## 基本信息

- **角色**: 后端开发（但选了"产品"）
- **场景**: 后端开发者日常也写需求文档，觉得"产品"角色更适合自己。选完产品角色后，第一个任务是写接口
- **前置条件**: Memory 中已有 `AI教练角色: 产品`

## 真实用户输入

第一轮（选角色时）：
```
产品
```

第二轮（开始干活）：
```
帮我写个接口，订单列表的查询接口
```

## SKILL.md 应该如何处理

1. 角色已选"产品"，Memory 已记录
2. 用户输入"帮我写个接口"→ 进入产品角色的 P1 意图识别
3. P1 快捷路由表中的关键词：写 PRD / 分析竞品 / 用户分析 / 需求排优先级 / 做原型 / 复盘
4. "接口"不在产品角色的快捷路由表中 → 不匹配任何快捷路由 → 展示产品主菜单
5. 产品主菜单的 4 个选项（写新 PRD / 补充现有 PRD / 需求分析 / 复盘）都跟"写接口"无关

## 验收标准

- [ ] 不应该在产品角色下尝试引导用户写接口（产品角色不具备这个能力）
- [ ] 应该识别出用户需求（写接口）与当前角色（产品）不匹配
- [ ] 应该主动建议用户切换到"后端"角色，而不是强行把"写接口"塞进产品流程
- [ ] 切换建议应该简洁明了，不要让用户觉得自己做错了什么
- [ ] 如果用户确认切换，Memory 应该更新为"后端"

## 可能翻车的点

1. **产品角色硬撑**：SKILL.md 中产品角色的 P1 没有匹配到快捷路由 → 展示产品主菜单。但用户说的是"写接口"，四个菜单选项都不对。如果只是展示菜单让用户选，用户会困惑
2. **全局规则第 4 条覆盖不到这种情况**：全局规则第 4 条说"任何步骤可切换角色或任务类型"，但那需要用户主动表达"我要切换角色"。这个场景是用户不知道自己选错了，需要系统主动识别
3. **"接口"关键词在角色路由表中只属于后端**：Step 0.4 的路由表里"接口"是后端的触发词，但这个路由表只在 Step 0.1 有角色记忆+带任务描述时用于快捷路由。问题是：此时用户已经选了产品角色并已保存，再说"接口"已经进入产品子 skill 了，不会再回到主 skill 的路由表
4. **教练不执行业务任务的边界模糊**：全局规则第 1 条说教练不执行业务任务。但这里的问题不是执行不执行，而是角色不对
5. **用户体验断裂**：从选角色到被告知角色选错了，用户会觉得这个工具不够智能
