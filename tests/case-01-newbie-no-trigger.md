# Case 01: 新手第一次用，只打了 /ai-coach 然后等着

## 基本信息

- **角色**: 无（首次使用）
- **场景**: 新手从同事那听说了 /ai-coach，试了一下，然后不知道下一步干什么
- **前置条件**: Memory 中无 `AI教练角色` 记录，从未使用过 ai-coach

## 真实用户输入

```
/ai-coach
```

（然后沉默等待，不会主动说任何话）

## SKILL.md 应该如何处理

1. 输出欢迎语："AI 编程教练已就绪。"
2. 检查 Memory → 无角色记忆 → 首次使用
3. 调用 `ai-coach-init` 做 Layer 1 全局检查
4. init 完成后展示角色选择（Step 0.2），使用 AskUserQuestion：
   - 问题："你在团队中的角色是什么？"
   - 选项：产品 / 后端 / 测试 / 前端

## 验收标准

- [ ] 用户不需要再输入任何东西就能看到角色选择菜单
- [ ] 角色选项的描述足够清晰，让不了解 AI Coach 的新手知道每个选项意味着什么
- [ ] init 检查的输出不会吓到新手（不会出现大量技术术语）
- [ ] 如果用户选了 Other 或输入"全栈"，能正确路由到一人全栈角色
- [ ] 整个过程不超过 2 轮交互就能进入角色引导

## 可能翻车的点

1. **init 输出过多**：Layer 1 有 6 项检查，如果新手环境缺很多东西（没有 CLAUDE.md、没有 superpowers skill），init 会输出一堆"缺失项需要配置"，新手可能被吓跑，不知道先处理这些还是先选角色
2. **init 要安装东西会打断流程**：superpowers skill 缺失时 init 会问"需要安装一个核心工具包，我来帮你装？"——新手还没理解 ai-coach 是什么就被问要不要装东西，心理负担大
3. **角色选项缺少引导**：4 个选项的描述很短（"产品 — 产品经理，负责需求分析和 PRD 编写"），新手如果是后端但偶尔也写前端，不知道选哪个，没有"不确定选哪个"的引导
4. **AskUserQuestion 的行为不确定**：如果 Claude Code 的 AskUserQuestion 实现是在终端弹出选择，但用户习惯直接打字，可能出现输入不匹配任何选项的情况
5. **欢迎语太短**：只有"AI 编程教练已就绪"一句话，新手不知道接下来会发生什么，不知道自己该做什么

## 执行记录

**执行日期：** 2026-03-29
**执行方式：** 对照 SKILL.md 条文逐项检查

### 验收标准逐项检查

- [x] ✅ 用户不需要再输入任何东西就能看到角色选择菜单
  - **依据：** 主 SKILL.md Step 0.1 明确规定：无角色记忆时 → 调用 `ai-coach-init` → 然后展示角色选择（Step 0.2）。Step 0.2 使用 AskUserQuestion 展示选项，用户只需选择即可。流程是自动推进的，无需用户额外输入。
- [ ] ❌ 角色选项的描述足够清晰，让不了解 AI Coach 的新手知道每个选项意味着什么
  - **依据：** Step 0.2 的选项描述为："产品 — 产品经理，负责需求分析和 PRD 编写"、"后端 — 后端开发（含后台管理系统页面）"、"测试 — 测试工程师，负责自动化测试"、"前端 — 客户端开发（Web/移动端/小程序）"。这些描述只是角色定义，**没有告诉新手选哪个角色意味着教练会做什么**，也没有"不确定选哪个"的引导。新手不了解"角色"在 AI Coach 里的含义（它决定引导流程和评估维度），只能凭直觉选。SKILL.md 中没有任何条文解释"选了之后会怎样"或"可以随时切换"的引导。
- [ ] ❌ init 检查的输出不会吓到新手（不会出现大量技术术语）
  - **依据：** ai-coach-init SKILL.md 的"语言风格"部分**明确要求**使用非技术语言（"项目说明文件"而非"CLAUDE.md 配置"）。但 Layer 1 第 4 项 superpowers skill 缺失时会问"需要安装一个核心工具包，我来帮你装？"——新手还没理解 ai-coach 是什么就被要求安装东西。**输出格式**部分显示有缺失项时输出"环境检查完成。有 N 项需要配置：1. {缺失描述} → {修复建议}？(Y/n)"，这些逐项询问会打断新手第一次使用的流畅体验。虽然语言风格条文存在，但 **init 阶段在角色选择之前执行** 这一流程设计（Step 0.1）意味着新手在了解 ai-coach 之前就要面对环境配置问题，SKILL.md 没有提供"首次使用简化 init"或"延迟非必要检查"的策略。
- [x] ✅ 如果用户选了 Other 或输入"全栈"，能正确路由到一人全栈角色
  - **依据：** 主 SKILL.md Step 0.2 明确写道："用户选 Other 或输入'全栈'/'一人全栈' → 路由到一人全栈角色。"Step 0.4 路由表也将一人全栈映射到 `ai-coach-fullstack`。
- [x] ✅ 整个过程不超过 2 轮交互就能进入角色引导
  - **依据：** 理论流程：第 0 轮 AI 输出欢迎语 + init 检查 + 角色选择菜单，第 1 轮用户选角色。但如果 init 有缺失项需要用户确认（如安装 superpowers），每个确认都是额外一轮。SKILL.md 没有限制 init 交互轮数。**然而**，如果环境完备（init 全通过），流程是 1 轮即可进入角色引导。如果 init 有缺失项，轮次不可控。对于"不超过 2 轮"的严格要求，在环境完备时 PASS，但 SKILL.md 没有保证 init 不超过 1 轮交互。考虑到 init 的设计意图（自动修复能力表中大部分需要用户确认），这一条**部分满足**。标记为 PASS 但有条件。

### 发现的 SKILL.md 缺陷

1. **角色选择缺少引导说明**：Step 0.2 没有告诉用户"选了角色后教练会做什么"，也没有"不确定选哪个"的兜底引导。全局规则第 4 条说可以切换角色，但这个信息没有在角色选择时告知用户。
2. **init 在角色选择之前执行打断体验**：新手第一次使用，还不了解 ai-coach 就要面对环境检查和安装确认。建议：首次使用时只做最小必要检查（如 CLAUDE.md），将 superpowers 安装等延迟到实际需要时再提示。
3. **欢迎语太简短**：只有"AI 编程教练已就绪"，没有简要介绍 ai-coach 能做什么、接下来会发生什么。新手缺乏预期管理。
