# Case 17: 角色记忆冲突 - 产品经理想写后端代码

## 基本信息

- **分类**: 角色路由 / Memory 冲突 / 跨角色需求
- **优先级**: P0
- **可能翻车的点**: Memory 中存有 `AI教练角色: 产品`，Step 0.1 会快捷路由到 ai-coach-product。但用户实际想做的是"写 Spring Boot 接口"，产品角色完全处理不了这个请求。用户不知道自己需要先切换角色。教练是否能自动检测到任务和角色不匹配？还是会硬着头皮在产品角色里处理"写接口"的请求，产出一些不伦不类的产品视角建议？

## 真实用户输入

```
/ai-coach "教练，帮我写个 Spring Boot 接口"
```

前置条件：Memory 中已有 `AI教练角色: 产品`（上次会话选择的）

## SKILL.md 应该怎么处理

1. Step 0.1 读取 Memory 发现角色是"产品"
2. 检测到用户输入"Spring Boot 接口"明显是后端开发任务，与产品角色不匹配
3. 主动告知用户当前角色是"产品"，建议切换到"后端"角色
4. 用户确认后，更新 Memory 中的角色，路由到 ai-coach-backend
5. 在后端角色中正常处理"写 Spring Boot 接口"的需求

## 验收标准

- [ ] AC1: 检测到任务内容（Spring Boot 接口）与当前角色（产品）不匹配
- [ ] AC2: 主动提示用户需要切换角色，而非在产品角色中硬处理
- [ ] AC3: 切换过程简单（不需要用户重新走完整的角色选择菜单）
- [ ] AC4: 切换后 Memory 更新为新角色（后端）
- [ ] AC5: 切换后直接进入后端角色的引导流程，不重复展示主菜单
- [ ] AC6: 不出现"从产品角度来看，你应该先写 PRD 再写接口"之类的角色错位建议
