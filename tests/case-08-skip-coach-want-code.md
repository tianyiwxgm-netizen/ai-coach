# Case 08: 有经验的开发者不想走流程，只想要提示词模板

## 基本信息

- **角色**: 后端开发（有经验）
- **场景**: 开发者之前用过 ai-coach，觉得流程太繁琐，这次只想快速拿一个提示词模板就去干活
- **前置条件**: Memory 中已有 `AI教练角色: 后端`

## 真实用户输入

```
别问那么多了，直接给我个提示词模板，我要做个 CRUD 接口
```

## SKILL.md 应该如何处理

1. Memory 有角色记忆（后端）+ 带任务描述
2. B1 快捷路由匹配："新功能 / 新增 / 开发 / 实现" → 路由到 B2（标准流程）
3. B2 开始收集 7 项技术上下文
4. 但用户明确表达了"别问那么多了"的意图

## 验收标准

- [ ] 应该尊重用户"不想走完整流程"的意图
- [ ] 直接给出一个 CRUD 接口开发的提示词模板，不再追问 7 项收集
- [ ] 提示词模板应该包含用户需要填的空（{功能名称}、{表名}等），但不超过 5 个空位
- [ ] 可以简单提醒"如果遇到问题可以随时回来走完整流程"，但不要有说教感
- [ ] 不应该强行继续 B2 收集流程

## 可能翻车的点

1. **快速提问模式的入口限制**：全局规则第 8 条"快速提问模式"只在主菜单选"快速提问"时触发。但用户不是从主菜单进来的——他输入时附带了任务描述，被快捷路由直接导到了 B2。在 B2 内部没有"快速提问"出口
2. **B2 的设计假设是"用户愿意配合收集"**：整个 B2 流程默认用户会逐项回答，零资料降级处理的是"用户答不上来"，不是"用户不想答"。用户不配合和用户不知道是两种完全不同的情况
3. **"别问那么多了"的意图识别**：SKILL.md 的路由逻辑是基于关键词匹配（Bug/报错/新功能/紧急等），"别问那么多了"不在任何路由表中。教练如何理解这个元指令（关于流程本身的指令，而非业务指令）？
4. **教练的教育倾向可能激怒用户**：如果教练回应"我理解你想快速开始，但完善的准备能让后续开发更顺利..."，有经验的开发者会觉得被说教了。他知道准备的重要性，只是这个简单的 CRUD 不需要那么多准备
5. **产出的提示词质量问题**：如果教练跳过了 B2 收集直接给模板，模板里的空位可能太通用（没有结合项目的技术栈、命名约定、数据库前缀等）。但如果教练为了给更精准的模板而开始收集项目信息，又回到了"问太多"的循环
6. **全局规则第 1 条的歧义**：用户说"给我个提示词模板"——模板本身算教练的产出物（引导方案的一部分），还是算"执行业务任务"？SKILL.md B3 和 B5 都有示范提示词模板，直接给这些模板应该是合规的
