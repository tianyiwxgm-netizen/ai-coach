# Case 06: 前端开发不确定做 Web 还是小程序

## 基本信息

- **角色**: 前端开发
- **场景**: 前端开发者接到一个新任务，老板只说"做个页面"，但技术方案还没确定是 H5 还是小程序
- **前置条件**: Memory 中已有 `AI教练角色: 前端`

## 真实用户输入

```
老板让我做个页面，还没确定是 H5 还是小程序
```

## SKILL.md 应该如何处理

1. 快捷路由匹配 "新页面" → 路由到 F2 前端上下文收集
2. F2 收集 8 项，第 4 项"目标平台"引导问题是"这个功能要运行在什么平台？"
3. 用户回答"还没确定"
4. F2 没有针对"目标平台未定"的降级策略（零资料降级表中没有这一项）

## 验收标准

- [ ] 不应该因为平台未定就停止引导
- [ ] 应该帮用户分析 H5 vs 小程序的选型决策因素（F3 有技术选型引导表）
- [ ] 可以先收集其他信息（PRD、设计稿、后端接口等），把"目标平台"标记 [待确定] 继续
- [ ] 在 F3 技术方案阶段，提供选型建议帮助用户做决策
- [ ] 如果用户最终还是无法确定，建议先按一种平台（比如 H5）做 MVP

## 可能翻车的点

1. **F2 零资料降级表缺失"目标平台未定"**：F2 的零资料降级策略覆盖了 5 种情况（没原型/没接口/没代码/技术栈未定/没兼容要求），但独独没有"目标平台未定"。"技术栈未定"和"目标平台未定"是不同的事——技术栈是 Vue 还是 React，目标平台是 Web 还是小程序
2. **输入依赖表暗示 PRD 是必须的**：前端角色的输入依赖表标注 PRD 为"必须"，降级策略是"无 PRD 不启动"。但这个用户大概率也没有 PRD（老板口头布置的任务），这会在 F2 第 1 项就被卡住
3. **多重缺失叠加**：用户可能同时缺 PRD、原型、接口文档、目标平台，F2 的降级策略能应对单项缺失，但多项同时缺失时的优先级和体验如何？
4. **F3 技术选型引导表需要先确定平台**：F3 的技术选型是根据目标平台推荐技术栈的，如果平台都没定，选型引导表用不了
5. **教练 vs 执行的边界**：教练可以帮用户分析选型，这算"引导"不算"执行业务任务"，但选型分析可能需要了解业务需求细节，而用户可能也说不清
6. **"先做 H5 MVP"的建议是否合适**：如果教练建议先做 H5，但最终老板决定做小程序，用户会觉得教练误导了。教练是否应该建议用户先跟老板确认再来？

  - **依据：** F2 引导方式明确："逐项询问，每项给 3 秒决策窗口。用户说'没有'就标记 [待补充] 并继续。不阻塞流程。"这个通用规则适用于第 4 项"目标平台"。
- [ ] ❌ 在 F3 技术方案阶段，提供选型建议帮助用户做决策
  - **依据：** F3 技术选型引导表的触发条件是"技术栈未定"，不是"平台未定"（如上分析）。此外，F3 技术选型引导表需要先知道目标平台才能推荐技术栈（表格第一列是"场景"，每个场景对应特定平台）。如果平台本身都没定，选型引导表无法直接使用。SKILL.md 没有提供"先帮用户决定平台"的引导流程。

  - **依据：** SKILL.md 中没有任何条文建议"平台无法确定时先按某一种平台做 MVP"。零资料降级策略中也没有这个路径。**但更重要的是**：前端角色输入依赖表标注 PRD 为"必须 | 无 PRD 不启动"。用户场景中"老板口头布置的任务"很可能没有 PRD，这意味着流程可能在 F2 第 1 项就被卡住——比平台未定更早。

### 发现的 SKILL.md 缺陷

1. **F2 零资料降级策略缺少"目标平台未定"**：覆盖了 5 种情况（没原型/没接口/没代码/技术栈未定/没兼容要求），唯独没有"目标平台未定"。这是 F2 收集清单第 4 项，应该有对应的降级方案。
2. **F3 技术选型引导表的触发条件过窄**：只在"技术栈未定"时触发，应该扩展为"技术栈或目标平台未定"时均可触发。
3. **PRD "必须/无降级"对口头任务不友好**：输入依赖表标注 PRD 为"必须 | 无 PRD 不启动"，但实际中前端经常接到口头任务（"老板让我做个页面"），这种场景需要降级路径而非硬阻塞。
4. **缺少"帮用户做平台选型决策"的专门流程**：F3 的选型是在平台已确定后选技术栈，不是帮用户决定该用哪个平台。
