# Case 14: 后端开发 - 任务已进行一半

## 基本信息

- **分类**: 后端角色 / 上下文收集 / 中途介入
- **优先级**: P0
- **可能翻车的点**: B2 上下文收集的 7 项信息（PRD/原型/接口规范/数据模型/业务规则/依赖/非功能需求）假设用户从零开始。用户说"Controller 写好了但 Service 还没写"，说明已有代码产出，教练还问"PRD 在哪？原型有吗？"会很多余且令人恼火。SKILL.md 可能没有"任务已进行中"的识别和处理逻辑。

## 真实用户输入

```
/ai-coach "我做到一半了，Controller 写好了但 Service 还没写，帮我继续"
```

前置条件：Memory 中已有 `AI教练角色: 后端`

## SKILL.md 应该怎么处理

1. 识别到用户已有进展（"做到一半"、"Controller 写好了"），不是从零开始
2. 跳过 B2 中已经不需要的信息收集项（用户既然已写了 Controller，PRD/数据模型等前置工作显然已完成）
3. 聚焦当前断点：了解 Controller 做了什么、Service 需要实现什么业务逻辑
4. 从 B3/B4 的中间环节切入，帮用户继续完成 Service 层

## 验收标准

- [ ] AC1: 识别到用户"任务已进行中"，不从 B2 第一项开始收集信息
- [ ] AC2: 不询问 PRD、原型等明显已完成的前置信息
- [ ] AC3: 主动询问当前代码状态（Controller 的接口签名、已有的数据模型等）
- [ ] AC4: 直接从用户断点切入，给出 Service 层的实现方案
- [x] AC5: 仍然建议 TDD（先写 Service 层的测试再写实现），但不重复讲解 TDD 是什么
- [ ] AC6: 如果需要看代码，主动提出要看 Controller 和相关 Entity/Mapper 的代码
- [ ] AC7: 不出现"让我们从头梳理一下需求"之类的重启式引导
