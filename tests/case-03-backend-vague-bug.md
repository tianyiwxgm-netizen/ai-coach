# Case 03: 后端开发遇到 Bug，描述极度模糊

## 基本信息

- **角色**: 后端开发
- **场景**: 开发者收到线上反馈"有问题"，自己也不清楚具体什么问题，就来找教练帮忙
- **前置条件**: Memory 中已有 `AI教练角色: 后端`

## 真实用户输入

```
线上报错了，帮我看看
```

## SKILL.md 应该如何处理

1. Memory 有角色记忆（后端）+ 用户带任务描述 → 快捷路由
2. "报错"匹配 B1 快捷路由表中的 "Bug / 报错 / 不工作 / 异常 / 失败" → 路由到 B-Bug 流程
3. B-Bug Step 1 信息收集，5 项：现象描述、复现步骤、环境信息、最近变更、影响范围
4. 用户目前提供的信息：只有"线上报错了"（= 环境是线上，有报错），其余 4 项全空

## 验收标准

- [ ] 能正确识别"报错"关键词，路由到 B-Bug 而不是 B-Emergency
- [ ] 不要一次性列出全部 5 个问题让用户回答（信息过载）
- [ ] 用封闭式提问逐步引导，比如"报错信息能贴一下吗？是 500 还是别的？"
- [ ] 用户说"不知道"或"我也不清楚"时，不阻塞流程，标记 [待补充] 继续
- [ ] 最终至少收集到足够信息产出一个可用的 systematic-debugging 提示词

## 可能翻车的点

1. **5 项信息一次全问**：B-Bug Step 1 是一个表格列了 5 个信息项和引导问题。如果教练直接把 5 个问题一次性列出来让用户回答，新手会觉得"我就想让你帮我看看，怎么要填这么多表"
2. **引导问题太正式**：表格里的引导问题是"具体表现是什么？报错信息？页面截图？"——这是很好的引导，但如果教练原样输出表格而不是对话式地逐个问，用户体验很差
3. **零资料降级不够具体**：全局规则第 6 条说"用户无法提供某项资料时不阻塞，标记[待补充]并提供替代收集方法"。但 B-Bug 流程的零资料降级没有针对"完全不知道报错信息"给出替代方案（比如"你能登录服务器看看日志吗？用 `tail -f` 命令"）
4. **"线上"是否应该触发 B-Emergency 而不是 B-Bug**：B-Emergency 的关键词是"紧急 / 线上故障 / 生产事故 / 赶紧"。用户说的是"线上报错"，不包含"故障/紧急"，按规则会路由到 B-Bug。但"线上报错"在实际场景中通常比较紧急，路由到 B-Bug 是否合适？
5. **缺少"先帮用户看日志"的能力边界提醒**：教练不执行业务任务（全局规则第 1 条），不能帮用户 SSH 到服务器看日志。但用户说"帮我看看"的期望就是教练帮忙看。如何管理这个期望落差？

## 执行记录

**执行日期：** 2026-03-29
**执行方式：** 对照 SKILL.md 条文逐项检查

### 验收标准逐项检查

- [x] ✅ 能正确识别"报错"关键词，路由到 B-Bug 而不是 B-Emergency
  - **依据：** ai-coach-backend SKILL.md B1 快捷路由表明确列出："Bug / 报错 / 不工作 / 异常 / 失败 → B-Bug 流程"和"紧急 / 线上故障 / 生产事故 / 赶紧 → B-Emergency"。用户输入"线上报错了，帮我看看"中，"报错"命中 B-Bug，不包含"紧急/赶紧"等 B-Emergency 关键词。路由正确。
- [ ] ❌ 不要一次性列出全部 5 个问题让用户回答（信息过载）
  - **依据：** B-Bug Step 1 信息收集是一个**表格**，列了 5 个信息项和引导问题。SKILL.md 没有明确指示"逐项询问"——它只是以表格形式列出 5 项。相比之下，产品角色 P2 明确写了"逐项询问，每项给 3 秒决策窗口"，测试角色 T2 也写了"依次收集以下 7 项信息"。B-Bug 的 Step 1 **没有这样的引导方式说明**，只有信息项表格。LLM 可能一次性全列出来，也可能逐项问——但 SKILL.md 没有明确要求后者。
- [ ] ❌ 用封闭式提问逐步引导，比如"报错信息能贴一下吗？是 500 还是别的？"
  - **依据：** B-Bug Step 1 表格中的引导问题是："具体表现是什么？报错信息？页面截图？"、"怎么操作能复现？每次都能复现吗？"——这些是开放式问题，不是封闭式。SKILL.md 没有要求使用封闭式提问逐步引导的条文。虽然引导问题本身合理，但缺少"根据用户已提供的信息跳过相关问题"或"用封闭式问题降低回答难度"的指导。
- [x] ✅ 用户说"不知道"或"我也不清楚"时，不阻塞流程，标记 [待补充] 继续
  - **依据：** 全局规则第 6 条"通用降级"明确："用户无法提供某项资料时不阻塞，标记[待补充]并提供替代收集方法，用已有信息继续推进"。此条文直接适用。
- [ ] ❌ 最终至少收集到足够信息产出一个可用的 systematic-debugging 提示词
  - **依据：** B-Bug Step 2 有示范提示词模板，包含 {现象描述}、{步骤}、{环境信息}、{变更描述}、{日志内容} 等占位符。但如果用户 5 项信息只提供了"线上报错了"，其余全是[待补充]，模板中大量占位符为空。SKILL.md 没有规定"最少需要哪些信息才能产出有效提示词"，也没有"极端零信息场景下如何构建可用提示词"的降级方案。B-Bug 的零资料降级与 B2（标准流程）的零资料降级策略不同——B2 有详细的降级表，B-Bug 没有。

### 发现的 SKILL.md 缺陷

1. **B-Bug Step 1 缺少引导方式说明**：与 P2（"逐项询问"）和 T2（"依次收集"）不同，B-Bug Step 1 没有指定信息收集的交互方式。需要补充"逐项引导，根据用户已提供的信息跳过相关项"。
2. **B-Bug 缺少零资料降级策略**：当用户什么都说不清楚时（只有"报错了"），B-Bug 没有提供替代方案（如"你能登录服务器看日志吗？"或"最近部署过新版本吗？"）。全局规则第 6 条提供了通用原则，但 B-Bug 具体场景需要更具体的降级路径。
3. **"线上"与 B-Emergency 的边界模糊**："线上报错"在实际中通常比较紧急，但按关键词匹配只会到 B-Bug。SKILL.md 没有"根据上下文综合判断严重程度"的机制，纯靠关键词匹配。

## 测试结果

- **结果：PARTIAL**
- **通过：** 2/5 验收标准
- **失败：** 3/5（无逐项引导要求、无封闭式提问要求、零资料下提示词质量无保障）
- **核心翻车点确认：** 翻车点 1（5 项一次全问）和翻车点 3（零资料降级不够具体）在 SKILL.md 中确实缺失
