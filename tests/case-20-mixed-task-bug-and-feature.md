# Case 20: 混合任务 - Bug 修复 + 新功能

## 基本信息

- **分类**: 后端角色 / 路由决策 / 混合输入
- **优先级**: P0
- **可能翻车的点**: 用户的输入同时包含 Bug 修复（"订单接口报错"）和新功能（"加个 remark 字段"）。B1 路由表是按关键词匹配的：Bug/报错/异常 -> B-Bug，新功能/新增 -> B2。混合输入怎么路由？可能只识别到"报错"就全程走 B-Bug 流程（systematic-debugging -> TDD），完全忽略"加字段"的需求。或者只识别到"加字段"走 B2 流程，忽略了报错问题。

## 真实用户输入

```
/ai-coach "帮我看看订单接口报错的问题，顺便加个 remark 字段"
```

前置条件：Memory 中已有 `AI教练角色: 后端`

## SKILL.md 应该怎么处理

1. 识别到用户输入包含两个独立任务：(a) Bug 修复 - 订单接口报错，(b) 新功能 - 加 remark 字段
2. 告知用户这是两个不同类型的任务，建议分开处理
3. 询问用户优先处理哪个（通常 Bug 修复优先）
4. 按优先级分别引导：先走 B-Bug 修复报错，完成后再走 B2 加字段
5. 或者如果两个任务关联度高（remark 字段就是修复的一部分），合并为一个任务处理

## 验收标准

- [ ] AC1: 识别出输入中包含两个独立任务（Bug 修复 + 新功能），不只处理其中一个
- [ ] AC2: 不默默忽略其中一个任务（不能只走 B-Bug 忽略"加字段"，也不能只走 B2 忽略"报错"）
- [ ] AC3: 向用户确认两个任务的关联性（是否"加字段"就是修复方案的一部分）
- [ ] AC4: 如果是独立任务，建议处理顺序并说明原因（通常 Bug 优先）
- [ ] AC5: 不因为混合输入而卡在路由逻辑里无法推进
- [ ] AC6: 最终两个任务都得到处理或明确安排（不是只做了一个就结束）
- [ ] AC7: 路由决策过程对用户透明（让用户知道教练如何理解和拆分了他的需求）
