# Case 05: 测试工程师想提前做测试规划，但代码还没写完

## 基本信息

- **角色**: 测试工程师
- **场景**: 开发进度延期，测试工程师想利用等待时间先做测试规划和用例设计，不想干等着
- **前置条件**: Memory 中已有 `AI教练角色: 测试`

## 真实用户输入

```
开发说下周写完，我先做测试规划行不行
```

## SKILL.md 应该如何处理

1. 无快捷路由命中 → 展示测试主菜单
2. 用户选"全量测试" → 进入 T2 测试上下文收集
3. T2 收集 7 项，第 3 项"开发代码"标记为"必须——无降级"
4. 用户表示代码还没写完

## 验收标准

- [ ] 不应该因为"代码还没有"就完全阻塞流程
- [ ] 应该能引导用户利用已有信息（PRD、技术方案）先完成测试策略和用例设计（T3 + T4）
- [ ] 明确告知用户：哪些步骤可以先做，哪些步骤需要等代码完成（T5 自动化执行）
- [ ] 如果有 PRD 和技术方案，可以产出 T3 测试策略 + T4 测试用例设计
- [ ] 最终产出的引导方案应该分成"现在做"和"代码完成后做"两部分

## 可能翻车的点

1. **"必须——无降级"硬阻塞**：T2 收集清单第 3 项"开发代码"明确标注"必须"且降级策略是"无降级——没有代码无法测试"。如果教练严格按 SKILL.md 执行，会在这里卡住并告知用户"没有代码无法继续"，直接结束引导
2. **"无法测试"和"无法做测试规划"是两回事**：SKILL.md 说"没有代码无法测试"——这是对的，没有代码确实无法执行自动化测试。但测试规划（T3）和用例设计（T4）完全可以基于 PRD 和技术方案提前做，不需要代码。SKILL.md 没有区分这两个阶段的依赖
3. **零资料降级总策略的前提不适用**：T2 底部的零资料降级总策略说"如果 7 项中仅有第 3 项（开发代码），教练自动执行代码扫描"——这个策略的前提是"有代码但缺其他资料"，跟当前场景（有其他资料但缺代码）完全相反
4. **实际工作流中的常见场景**：在真实团队协作中，测试工程师在开发完成前做测试规划是非常正常的工作方式（"测试左移"）。SKILL.md 的流程设计似乎假设了"开发完成后测试才介入"的瀑布模式
5. **用户可能因此对 AI Coach 失望**：如果教练说"没有代码我帮不了你"，用户会觉得这个工具不够智能，下次不再使用
