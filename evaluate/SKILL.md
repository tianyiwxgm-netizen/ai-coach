# AI 编程教练 — 评估模式

本 Skill 由 ai-coach 主 skill 在用户选择"复盘总结"时调用。独立运行完整的评估流程。

**启动时输出：** "正在启动评估模式，开始分析你的 Claude Code 使用记录..."

## Step 9: 角色路由（评估入口）

从 Memory 中查找角色标识（格式：`AI教练角色: {角色名}`）。

```bash
MEMORY_DIR="$HOME/.claude/projects"
grep -r "AI教练角色" "$MEMORY_DIR"/*/memory/MEMORY.md 2>/dev/null | head -5
```

| Memory 中的角色 | 路由目标 | 评估维度集 |
|----------------|---------|-----------|
| 一人全栈 | Step 13-A | 8 维（保留现有微调） |
| 产品 | Step 13-B | 8 维（产品专属） |
| 后端 | Step 13-C | 8 维（后端专属） |
| 测试 | Step 13-D | 8 维（测试专属） |
| 前端 | Step 13-E | 8 维（前端专属） |
| **无角色记忆** | Step 13-A | **默认"一人全栈"** |

确定角色后，记录 `current_role` 变量，在后续 Step 13-16 中使用。

## Step 10: 评估模式入口

### 10.1 分析范围选择

**使用 AskUserQuestion：**

问题："你想分析哪些会话记录？"
选项：
- 今天的会话 — 只分析今天的使用记录
- 最近一周 — 分析过去 7 天的会话
- 全部历史 — 分析所有会话记录（可能较慢）

### 10.2 会话文件定位与分析

会话记录存储在：`~/.claude/projects/` 目录下，按项目路径组织。

**使用 analyze_sessions.py 批量分析：**

```bash
SCRIPT_DIR=$(cd "$(dirname "$(find ~/.claude/skills -name 'analyze_sessions.py' -path '*ai-coach*' 2>/dev/null | head -1)")" 2>/dev/null && pwd)
[ -z "$SCRIPT_DIR" ] && SCRIPT_DIR="$HOME/.claude/skills/ai-coach-evaluate"

# 今天
python3 "$SCRIPT_DIR/analyze_sessions.py" --dir ~/.claude/projects/ --days 1 --max 30
# 最近一周
python3 "$SCRIPT_DIR/analyze_sessions.py" --dir ~/.claude/projects/ --days 7 --max 30
# 全部历史
python3 "$SCRIPT_DIR/analyze_sessions.py" --dir ~/.claude/projects/ --max 50
```

脚本输出 JSON 包含：session_count、total_turns、skill_usage、tool_usage、plan_mode_count、test_command_count 等全部信号，可直接用于 Step 13 评分。

### 10.3 历史报告加载

```bash
HISTORY_FILE="{当前项目根目录}/quality/ai-coach/history.json"
[ -f "$HISTORY_FILE" ] && cat "$HISTORY_FILE"
```

如果存在历史数据，提取**同角色**的上次评分用于学习成长维度对比（不同角色不做横向对比）。

### 10.4 数据充分度判断

| 数据量 | 判定 | 处理方式 |
|--------|------|---------|
| 0 个会话 | 无数据 | 友好提示"还没有找到会话记录"，建议先完成几个任务再来，直接结束 |
| 1-2 个会话 | 数据不足 | 告知"数据量较少，仅供参考"，正常评估标注置信度"低" |
| 3-9 个会话 | 基本充分 | 正常评估，置信度"中" |
| 10+ 个会话 | 充分 | 正常评估，置信度"高" |

**置信度格式：** `85/100 (置信度: 高)`

## Step 11: 行为信号提取

Step 10.2 中 analyze_sessions.py 已输出全部汇总信号。

### 11.1 通用信号字段

| 字段 | 说明 | 用于 |
|------|------|------|
| `skill_usage` | 各 Skill 调用次数 | SOP 合规、方法论内化 |
| `tool_usage` | 各工具调用次数 | 工具成熟度 |
| `plan_mode_count` | Plan Mode 使用次数 | SOP 合规 |
| `test_command_count` | 测试命令执行次数 | TDD 与质量 |
| `git_commit_count` | Git 提交次数 | 提交质量 |
| `error_recovery_tools` | 出错后使用的工具 | 提示词交互 |
| `avg_user_msg_length` | 平均用户消息长度 | 提示词交互 |
| `user_msg_length_distribution` | 消息长度分布 (short/medium/long) | 提示词交互 |
| `claude_md_refs` | CLAUDE.md 引用次数 | 上下文工程 |
| `parallel_agents` | 并行 Agent 次数 | 工具成熟度 |
| `user_messages_sample` | 采样用户消息（最多 20 条） | 业务理解、方法论 |
| `per_session` | 每个会话摘要 | 学习成长 |

### 11.2 角色专属信号字段

| 字段 | 说明 | 适用角色 |
|------|------|---------|
| `verification_skill_usage` | verification-before-completion 次数 | 全角色 |
| `prd_references` | 引用 PRD/需求文档次数 | 产品、测试 |
| `prototype_outputs` | 产出 HTML 原型次数 | 产品 |
| `test_report_outputs` | 产出测试报告次数 | 测试 |
| `mock_api_usage` | 使用 mock 接口次数 | 前端 |
| `screenshot_count` | 截图验证次数 | 前端、测试 |
| `device_test_count` | 设备测试次数 | 前端 |

评估时按 `current_role` 过滤使用，无需额外合并步骤。

## Step 12: 上下文工程检查

使用 Glob 和 Read 检查学员的上下文工程基础设施：

| 检查项 | 检查方式 | 评分影响 |
|--------|---------|---------|
| CLAUDE.md 是否存在 | Glob `**/CLAUDE.md` | 上下文工程 |
| CLAUDE.md 内容完整度 | Read 后检查：快速命令、技术栈、架构约定、测试约定 | 上下文工程 |
| Memory 配置 | `~/.claude/projects/*/memory/MEMORY.md` | 上下文工程 |
| Skills 数量 | `~/.claude/skills/*/SKILL.md` 或 `skill.md` | 上下文工程 |
| Hooks 配置 | `.claude/settings.json` 中的 hooks | 上下文工程 |

## Step 13: 角色化评分

基于 Step 11-12 信号数据，根据 `current_role` 路由到对应评估维度集。

### ========== Step 13-A: 一人全栈（默认） ==========

#### A1: SOP 流程合规度（权重 18%）

**评分信号：** skill_usage 中 brainstorm/write-plan/execute-plan, plan_mode_count, 会话中先规划再编码的模式

**评分规则：**
- 0-20：无任何规划行为，所有会话直接写代码
- 21-40：偶尔使用 Plan Mode，但不用规划 Skill
- 41-60：使用了规划 Skill，但不是每次都用
- 61-80：大部分功能开发遵循 brainstorm -> plan -> execute 流程
- 81-100：稳定执行完整 SOP，包含审查和持续改进

**计算公式：** 规划率 = (brainstorm + write-plan + plan_mode) / 总会话数。>=80%->81-100, >=50%->61-80, >=20%->41-60, >0->21-40, 0->0-20

#### A2: 上下文工程质量（权重 13%）

**评分信号：** CLAUDE.md 存在/完整度, Memory, Skills 数量, Hooks, claude_md_refs

**评分规则：**
```
基础分 = 0
if CLAUDE.md 存在: +20
  if 有快速命令: +10; 有技术栈: +10; 有架构约定: +10; 有测试约定: +10
if Memory 配置: +15
  if Memory 有项目上下文记录: +10; 跨会话引用历史记录: +10
if Skills >= 3: +10; >= 5: +15
if Hooks 配置: +10
```

#### A3: 提示词与交互质量（权重 13%）

**评分信号：** avg_user_msg_length, user_msg_length_distribution, 提示词含目标/约束/验收标准, error_recovery_tools

**评分规则：**
- 平均长度 < 20 字：过于简短（-20）
- 平均长度 20-200 字：合理范围
- 平均长度 > 500 字：过于冗长
- 出错后换策略（换不同工具）：+20
- 出错后盲目重试（同一工具）：-10

#### A4: TDD 与质量意识（权重 18%）

**评分信号：** test_command_count, 测试在代码修改前/后, Red-Green-Refactor 模式

**评分规则：**
- 0-20：从未运行测试
- 21-40：偶尔运行测试但在代码之后
- 41-60：经常运行测试
- 61-80：有先测试后实现的模式
- 81-100：稳定 TDD（Red-Green-Refactor）

#### A5: 工具使用成熟度（权重 8%）

**评分信号：** tool_usage 分布, parallel_agents, git_commit_count/总会话数

**评分规则：**
- 0-20：只用 Bash
- 21-40：Bash + Read
- 41-60：Bash + Read + Grep/Glob
- 61-80：5+ 种工具 + 并行 Agent
- 81-100：工具组合熟练 + 并行 + Plan Mode + 频繁提交

#### A6: 业务理解能力（权重 8%）

**评分信号：** 消息中业务术语 vs 技术术语比例, 业务背景/用户场景描述, 业务导向 Skill 使用

**评分规则（AI 定性评估）：**
- 0-20：只说"帮我改这个函数"，从不提业务背景
- 21-40：偶尔提到功能目的，缺乏用户场景
- 41-60：偶尔提到用户场景，多数技术视角
- 61-80：多数需求包含业务背景和用户价值
- 81-100：稳定业务视角，主动使用业务导向 Skill

#### A7: 学习成长轨迹（权重 8%）

**评分信号：** 历史报告评分变化（同角色）, 近期 vs 早期行为, 新 Skill 学习

**评分规则：**
- 有历史报告：对比上次同角色评分，进步+10以上->高分；停滞->中分；退步->低分
- 无历史报告：基于会话时间线前后半段对比

#### A8: 方法论内化度（权重 14%）

**评分信号：** ailearn 五大原则体现, 人机分工意识, 复杂度阶梯思维, 评估驱动习惯

**评分规则（AI 定性评估）：**
- 0-20：完全没有方法论意识，把 AI 当搜索引擎
- 21-40：偶尔体现方法论思维，不稳定
- 41-60：部分原则已内化（如 TDD），其他缺失
- 61-80：多数原则已内化，能自觉应用
- 81-100：方法论已成本能，灵活组合运用

**关键行为指标：** 注意力预算(信噪比高), 简单优先(不过度工程化), 数据驱动(测试/指标验证), 人机分工(架构自己做执行交AI), 系统思维(关注工作流)

### ========== Step 13-B: 产品角色 ==========

#### B1: 历史资产利用（15%）
**信号：** prd_references, 消息中提及现有系统/历史数据/现有接口, 资产盘点行为
**评分：**
- 0-20：从零写需求，完全忽略现有系统
- 21-40：偶尔提及现有功能，未系统盘点
- 41-60：引用部分历史文档或代码，但不全面
- 61-80：系统盘点现有资产（文档+代码+DB），需求建立在现有基础上
- 81-100：每次需求先做资产盘点，主动发现可复用模块

#### B2: 业务分析深度（15%）
**信号：** 消息中商业目标/用户画像/竞品分析, brainstorming/jobs-to-be-done 使用, 量化指标
**评分：**
- 0-20：仅描述功能点，无业务背景
- 21-40：简单业务说明，缺用户洞察
- 41-60：有用户场景，缺数据支撑
- 61-80：用户分析+商业目标+量化指标
- 81-100：完整分析（用户+商业+竞品+数据），有优先级依据

#### B3: PRD 完整度（20%）
**信号：** prd-development Skill 使用, PRD 含用户故事/验收标准/字段定义/接口契约/异常处理
**评分：**
- 0-20：无 PRD，需求仅口头描述
- 21-40：简单需求文档，缺验收标准和字段
- 41-60：结构化 PRD，含基本用户故事和验收标准
- 61-80：含用户故事+验收+字段定义+接口契约
- 81-100：AI 可执行标准 PRD，后端可直接据此编码

#### B4: 原型交付率（15%）
**信号：** prototype_outputs, 可交互原型/线框图, 原型与 PRD 对应关系
**评分：**
- 0-20：从未产出原型
- 21-40：偶尔页面草图描述，无可交互产物
- 41-60：部分需求有 HTML 原型或 Mermaid 流程图
- 61-80：多数需求有可交互原型，页面间可跳转
- 81-100：每个需求完整原型+流程图，与 PRD 一一对应

#### B5: 需求沟通质量（10%）
**信号：** 沟通提纲准备, 与开发/业务方对齐, 交接引导完整度
**评分：**
- 0-20：不考虑下游理解
- 21-40：偶尔考虑实现难度
- 41-60：基本交接意识但不系统
- 61-80：准备了交接材料和沟通提纲
- 81-100：系统化需求管理，下游澄清<2轮

#### B6: Skill 使用熟练度（10%）
**信号：** skill_usage 中 prd-development/user-story/jobs-to-be-done/brainstorming 频率
**评分：**
- 0-20：从未使用 Skill
- 21-40：偶尔 brainstorming
- 41-60：2-3 种但不系统
- 61-80：正确环节用正确 Skill
- 81-100：Skill 成为工作流自然组成

#### B7: 上下文工程（10%）
**信号：** Step 12 检查结果
**评分：** CLAUDE.md 存在+25, 含产品约定+15, Memory+20, Memory 有产品决策记录+10, 跨会话引用+10, Skills>=3+15/>=5+20, docs/有组织+10, Hooks+5

#### B8: 学习成长（5%）
**评分规则同 A7。**

### ========== Step 13-C: 后端角色 ==========

#### C1: TDD 合规度（20%）
**信号：** test_command_count, 测试在代码前/后, TDD Skill 使用, Red-Green-Refactor
**评分：**
- 0-20：从未运行测试
- 21-40：偶尔测试，在代码之后（验证式）
- 41-60：经常测试，偶尔先写测试
- 61-80：多数先写测试再实现，使用 TDD Skill
- 81-100：稳定 TDD，Bug 修复也先写复现测试

#### C2: 技术方案质量（15%）
**信号：** 编码前方案(brainstorming+writing-plans), 方案含架构/表/接口/影响/测试策略
**评分：**
- 0-20：无方案直接写码
- 21-40：简单 Plan Mode 不系统
- 41-60：有表设计和接口设计
- 61-80：含架构分析+影响范围+测试策略
- 81-100：完整方案经 brainstorming，有风险评估

#### C3: 自测交付标准（20%）
**信号：** verification_skill_usage, 单元+集成+e2e 覆盖, 构建通过
**评分：**
- 0-20：写完直接交付，无自测
- 21-40：偶尔跑单测
- 41-60：有单测和基本集成测试
- 61-80：单元+集成+页面+verification Skill
- 81-100：完整自测（单元+集成+e2e+构建验证），零缺陷交付

#### C4: SOP 流程合规（15%）
**信号同 A1。**
**评分：**
- 0-20：所有会话直接写码
- 21-40：偶尔 Plan Mode
- 41-60：部分功能遵循规划
- 61-80：大部分遵循 brainstorm->plan->execute
- 81-100：稳定 SOP，含 Bug 的 debugging->TDD 流程

#### C5: 代码提交质量（10%）
**信号：** git_commit_count/会话数, 小步提交模式, 提交消息质量
**评分：**
- 0-20：很少提交或一次大 commit
- 21-40：有提交但不频繁
- 41-60：每功能有提交，粒度偏大
- 61-80：小步提交，每 Phase 至少一次
- 81-100：功能点即提交，消息清晰便于回滚

#### C6: 上下文工程（10%）
**信号：** Step 12 检查结果
**评分：** CLAUDE.md 存在+25, 含快速命令+10/技术栈+10/测试约定+10, Memory+15, Memory 有技术方案记录+10, 跨会话引用+10, Skills>=3+10/>=5+15, Hooks+10

#### C7: Bug 修复规范（5%）
**信号：** systematic-debugging 使用, Bug 修复先写复现测试, 根因分析过程
**评分：**
- 0-20：遇 Bug 直接改码，无分析
- 21-40：简单排查不系统
- 41-60：偶尔用 debugging Skill
- 61-80：多数走 debugging->TDD
- 81-100：稳定根因分析->复现测试->修复->验证

#### C8: 学习成长（5%）
**评分规则同 A7。**

### ========== Step 13-D: 测试角色 ==========

#### D1: 测试用例设计（20%）
**信号：** 消息中用例设计(ID/场景/输入/输出), 正常+异常+边界覆盖, 优先级排序
**评分：**
- 0-20：无用例设计，直接写自动化
- 21-40：简单用例仅正常场景
- 41-60：正常+异常覆盖，缺边界
- 61-80：三层覆盖（正常+异常+边界），有优先级
- 81-100：结构化用例关联 PRD 验收标准，可追溯

#### D2: 自动化比例（20%）
**信号：** test_command_count, Playwright 使用, 手动 vs 自动化比例
**评分：**
- 0-20：全手动测试
- 21-40：基本单测，e2e 全手动
- 41-60：单元+集成自动化，e2e 部分
- 61-80：三层都有自动化
- 81-100：高覆盖 + Playwright 三代理 + CI

#### D3: PRD 覆盖率（15%）
**信号：** prd_references, 用例关联 PRD 编号, 需求覆盖矩阵
**评分：**
- 0-20：测试与 PRD 无关联
- 21-40：偶尔对照 PRD
- 41-60：部分用例关联需求
- 61-80：多数关联有覆盖率统计
- 81-100：覆盖矩阵完整，每个验收标准对应用例

#### D4: 测试金字塔合理性（10%）
**信号：** 单元 vs 集成 vs e2e 比例
**评分：**
- 0-20：只一种类型
- 21-40：两种但比例失衡
- 41-60：三层比例不合理
- 61-80：金字塔基本合理（单元>集成>e2e）
- 81-100：健康金字塔，各层职责清晰

#### D5: 测试报告规范度（10%）
**信号：** test_report_outputs, 报告含概览/通过率/失败详情/缺陷清单
**评分：**
- 0-20：无报告
- 21-40：口头汇报结果
- 41-60：简单报告缺结构化数据
- 61-80：标准化报告（概览+通过率+失败+缺陷）
- 81-100：含趋势分析+PRD 覆盖率+发布建议

#### D6: Skill 使用熟练度（10%）
**信号：** skill_usage 中 test-driven-development/webapp-testing/verification
**评分：**
- 0-20：从未用 Skill
- 21-40：偶尔 TDD
- 41-60：TDD + webapp-testing
- 61-80：正确环节正确 Skill
- 81-100：Skill 组合熟练，测试工作流 Skill 驱动
**治具加分：** CLAUDE.md 存在+10, 含测试约定+5, Memory 有测试策略记录+10, 跨会话引用+10, Skills>=3+5, Hooks+5

#### D7: 缺陷有效性（10%）
**信号：** 缺陷含复现步骤, 描述含环境/步骤/实际/期望, 优先级区分
**评分：**
- 0-20：描述模糊（"功能不好用"）

### ========== 综合评分（所有角色通用） ==========

```
总分 = SUM(维度评分 * 权重)
等级 = L1(0-30) | L2(31-50) | L3(51-70) | L4(71-85) | L5(86-100)
```

| 等级 | 描述 |
|------|------|
| L1 入门 | 刚开始使用 AI 编程，需要全面引导 |
| L2 基础 | 掌握基本操作，但缺乏系统方法 |
| L3 进阶 | 有方法论意识，部分实践已形成习惯 |
| L4 熟练 | 方法论内化，能高效利用 AI 编程 |
| L5 专家 | 方法论已成本能，能指导他人，持续创新工作流 |

### Step 14.1: 用户反馈处理

用户看到评分后可能有异议，按以下方式处理：

- **质疑某个维度评分**："这个维度评的是 {评分信号}，你的会话中 {具体数据}。如果你有补充信息，我可以调整。"
